# 考试场景详解（6大情景）

> 考试从以下6个场景中随机抽取**4个**出题。每个场景覆盖多个 Domain。

---

## 场景 1：客户支持解决智能体

**背景：**  
使用 Claude Agent SDK 构建一个客户支持解决智能体。处理高歧义性请求（退货、账单争议、账户问题），通过自定义 MCP 工具访问后端系统，目标是实现 80%+ 的首次接触解决率，同时知道何时升级到人工。

**可用 MCP 工具：**
- `get_customer` — 验证客户身份
- `lookup_order` — 查询订单详情
- `process_refund` — 处理退款
- `escalate_to_human` — 升级到人工

**主要考查域：** Domain 1、Domain 2、Domain 5

考点：
- 退款前必须先验证客户身份（程序化强制执行 vs 提示引导）
- `PostToolUse` 钩子规范化不同工具返回的数据格式
- 结构化交接协议（升级给人工时的摘要内容）
- 工具描述清晰度（`get_customer` vs `lookup_order` 的区分）

---

## 场景 2：使用 Claude Code 进行代码生成

**背景：**  
使用 Claude Code 加速软件开发，包括代码生成、重构、调试和文档编写。通过自定义斜杠命令、CLAUDE.md 配置集成到开发工作流，理解何时使用 Plan Mode vs 直接执行模式。

**主要考查域：** Domain 3、Domain 5

考点：
- CLAUDE.md 配置的作用范围和结构
- Plan Mode 适用场景（高风险操作前预览）
- 自定义斜杠命令的定义方式
- Hooks 触发时机（PostToolUse、PreToolUse）

---

## 场景 3：多智能体研究系统

**背景：**  
使用 Claude Agent SDK 构建多智能体研究系统。协调智能体委托给专业子智能体：一个搜索网络、一个分析文档、一个综合发现、一个生成报告。研究特定主题并产出综合引用报告。

**智能体架构：**
```
协调智能体
├── 网络搜索子智能体
├── 文档分析子智能体
├── 综合发现子智能体
└── 报告生成子智能体
```

**主要考查域：** Domain 1、Domain 2、Domain 5

考点：
- 任务分解过窄导致覆盖不完整（官方样题第7题）
- 子智能体上下文隔离（必须显式传递）
- 并行 vs 串行子智能体调用
- 结构化上下文传递（保留来源归因）
- 上下文窗口管理（长研究报告）

---

## 场景 4：Claude 驱动的开发者生产力工具

**背景：**  
使用 Claude Agent SDK 构建开发者生产力工具。智能体帮助工程师探索陌生代码库、理解遗留系统、生成样板代码、自动化重复任务。使用内置工具（Read、Write、Bash、Grep、Glob）并集成 MCP 服务器。

**主要考查域：** Domain 2、Domain 3、Domain 1

考点：
- 内置工具的适用场景（Read/Write/Bash/Grep/Glob 各自的适用场景）
- MCP 服务器配置（`.mcp.json` vs `~/.claude.json`）
- 动态任务分解（探索遗留代码库）
- 会话状态管理与恢复

---

## 场景 5：CI/CD 中的 Claude Code

**背景：**  
将 Claude Code 集成到 CI/CD 流水线。系统运行自动代码审查、生成测试用例、对 PR 提供反馈。需要设计提供可操作反馈并最小化误报的提示。

**主要考查域：** Domain 3、Domain 4

考点：
- 提示链（prompt chaining）设计（代码审查的分步骤）
- Few-shot 示例的作用（展示期望的反馈格式）
- JSON Schema 约束输出（结构化审查结果）
- 减少误报的提示设计技巧
- Claude Code 在非交互/headless 模式下的配置

---

## 场景 6：结构化数据提取系统

**背景：**  
使用 Claude 构建结构化数据提取系统。系统从非结构化文档中提取信息、使用 JSON Schema 验证输出、保持高准确率，并能优雅处理边缘情况与下游系统集成。

**主要考查域：** Domain 4、Domain 5

考点：
- JSON Schema 定义与输出约束
- Few-shot 示例设计（展示提取格式）
- 边缘情况处理（缺失字段、格式异常）
- 长文档分块处理策略
- 置信度标记与人工审核触发
