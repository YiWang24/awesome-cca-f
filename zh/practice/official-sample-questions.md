# 官方样题精解（12题）

> 来源：CCA-F 官方考试指南 v0.1  
> **重要提示**：这12题是官方公开的样题，考试中可能出现相同或相似的题目，务必理解每道题的核心考点。

## 场景一：客户支持解决智能体

> **背景**：使用 Claude Agent SDK 构建客户支持智能体，工具包括 `get_customer`、`lookup_order`、`process_refund`、`escalate_to_human`

---

### Q1 — 程序化约束 vs 提示引导

生产数据表明，在 **12%** 的案例里，智能体完全跳过 `get_customer`，只根据客户口头提供的姓名调用 `lookup_order`，偶尔导致账户误识别和错误退款。什么修改最能有效解决这个可靠性问题？

A. 增加程序化前置条件：在 `get_customer` 返回已验证客户 ID 之前，阻止 `lookup_order` 和 `process_refund`  
B. 在 system prompt 中加强说明：任何订单相关操作前都必须先通过 `get_customer` 完成客户验证  
C. 增加 few-shot，展示智能体始终先调用 `get_customer`  
D. 加一个路由分类器，根据请求类型只启用合适工具子集

> **答案：A**  
> **核心考点**：程序化约束（确定性）vs 提示引导（概率性）  
> **解析**：当关键业务逻辑要求固定顺序时，程序化约束能提供确定性保障。B 和 C 只有概率性合规，有非零失败率，不足以应对有财务后果的场景。

---

### Q2 — 工具描述质量

日志显示，当用户询问订单问题（如"帮我查订单 #12345"）时，智能体经常调用 `get_customer`，而不是 `lookup_order`。两个工具都只有极简描述，且接受相似标识符格式。**最有效的第一步**是什么？

A. 在系统提示中加入 5 到 8 个 few-shot 示例  
B. 扩展工具描述，加入输入格式、示例查询、边界与适用场景  
C. 在每轮前解析用户输入并预先选定工具  
D. 合并成一个 `lookup_entity` 工具

> **答案：B**  
> **核心考点**：工具选择错误时，第一步改工具描述  
> **解析**：工具描述是工具选择的首要依据，根因是描述质量不足。C（路由层）是重量级方案，不是第一步。

---

### Q3 — 升级校准

智能体首次接触解决率只有 **55%**，远低于 80% 目标。日志显示：它会升级一些本来很简单的情况，却尝试自主处理需要政策例外的复杂情况。改进升级校准的**最有效方式**是什么？

A. 在 system prompt 中加入明确升级标准，并用 few-shot 展示何时升级、何时自主处理  
B. 让智能体先自报一个 `1-10` 置信分，低于阈值则自动转人工  
C. 训练单独分类器预测哪些请求需要升级  
D. 用情绪分析在负面情绪超阈值时自动升级

> **答案：A**  
> **核心考点**：升级标准要明确，置信分和情绪分析不是可靠升级指标  
> **解析**：根因是升级边界不清楚，最直接的修正是加明确标准和示例。

## 场景二：使用 Claude Code 进行代码生成

---

### Q4 — 斜杠命令位置

你要创建一个自定义 `/review` 命令，用于运行团队标准代码审查清单，所有 clone 仓库的开发者都能自动使用。这个命令文件应放在哪里？

A. 项目仓库中的 `.claude/commands/`  
B. 每位开发者本机的 `~/.claude/commands/`  
C. 项目根目录的 `CLAUDE.md`  
D. `.claude/config.json`

> **答案：A**  
> **核心考点**：项目级命令 vs 用户级命令  
> **解析**：`.claude/commands/` 通过版本控制共享给所有团队成员；`~/.claude/commands/` 仅个人使用。

---

### Q5 — Plan Mode 判断

你被要求把团队的单体应用拆成微服务。这会影响几十个文件，并涉及服务边界和模块依赖的决策。你应该怎么做？

A. 进入 Plan Mode，先探索代码库、理解依赖、设计实现方案，再开始修改  
B. 直接执行，边改边让实现暴露自然边界  
C. 直接执行，但给出一大段超详细 upfront instructions  
D. 先直接执行，遇到复杂度再切到 Plan Mode

> **答案：A**  
> **核心考点**：大规模架构变更必须先进入 Plan Mode  
> **解析**：影响几十个文件、涉及架构决策 → 典型的 Plan Mode 场景。

---

### Q6 — 路径特定规则

代码库有不同区域使用不同约定：React 组件、API 处理器、数据库模型。测试文件分散在各处，但都应遵守统一测试规范。要让 Claude **自动套用正确规则**，最可维护的方式是什么？

A. 在 `.claude/rules/` 中创建带 YAML frontmatter 的规则文件，用 glob 模式按路径匹配  
B. 把所有规范都写在根 `CLAUDE.md` 里，让 Claude 自己推断  
C. 为每种代码类型创建 skill  
D. 每个子目录放一个 `CLAUDE.md`

> **答案：A**  
> **核心考点**：测试文件散落各处 → 用 glob 模式而非目录级 CLAUDE.md  
> **解析**：`**/*.test.tsx` 类的 glob 模式可跨目录匹配，目录级 CLAUDE.md 无法处理这种情况。

## 场景三：多智能体研究系统

---

### Q7 — 任务分解过窄

研究主题是"AI 对创意产业的影响"。各子智能体运行正常，但最终报告只覆盖视觉艺术，完全漏掉音乐、写作和电影制作。协调者把任务拆成了：AI in digital art creation、AI in graphic design、AI in photography。**最可能的根因**是什么？

A. 综合子智能体不会识别覆盖缺口  
B. 协调者的任务拆解过窄，导致子任务没有覆盖主题所有相关领域  
C. 网页搜索子智能体查询不够全面  
D. 文档分析子智能体过滤了非视觉艺术来源

> **答案：B**  
> **核心考点**：多智能体系统覆盖不全 → 先看协调者的任务分解  
> **解析**：问题在协调者，不在子智能体。分解任务时覆盖面过窄是根因。

---

### Q8 — 结构化错误传播

网页搜索子智能体在研究复杂主题时超时。你需要设计这次失败信息如何回传给协调者。哪种错误传播方式**最有利于智能恢复**？

A. 返回结构化错误上下文：失败类型、尝试的查询、部分结果、潜在替代方案  
B. 子智能体内部自动指数退避重试，耗尽后只回一个通用状态  
C. 把超时吞掉并返回空结果且标记成功  
D. 直接把异常抛到顶层并终止整个流程

> **答案：A**  
> **核心考点**：错误传播要包含足够信息让协调者做恢复决策  
> **解析**：结构化错误上下文（失败类型 + 已尝试查询 + 部分结果 + 替代方案）让协调者能选择重试、修改策略或带部分结果继续。

---

### Q9 — 工具范围优化

综合子智能体在整合结果时常需核查简单事实。现在每次都要把控制权交回协调者，再转给搜索子智能体，导致 2-3 次往返，延迟增加 40%。评估发现 **85%** 是简单事实核查，**15%** 才需要深度研究。最佳优化方案是什么？

A. 给综合子智能体一个受限的 `verify_fact` 工具处理简单核查，复杂核查仍经协调者委派  
B. 综合子智能体把所有核查需求攒到最后一起发回协调者  
C. 给综合子智能体全部搜索工具  
D. 让搜索子智能体一开始就缓存更多上下文

> **答案：A**  
> **核心考点**：受限工具 + 按复杂度路由，不是给所有工具或不给工具  
> **解析**：给综合子智能体一个受限的 `verify_fact` 工具（处理简单85%），复杂验证（15%）仍走协调者委派路径。

## 场景四：CI/CD 中的 Claude Code

---

### Q10 — 非交互模式

你的流水线运行：
```bash
claude "Analyze this pull request for security issues"
```
结果作业无限挂起，日志显示 Claude Code 在等待交互输入。正确做法是什么？

A. 加 `-p`：`claude -p "Analyze this pull request for security issues"`  
B. 设环境变量 `CLAUDE_HEADLESS=true`  
C. 把 stdin 重定向到 `/dev/null`  
D. 加 `--batch`

> **答案：A**  
> **核心考点**：CI 中 Claude Code 必须用 `-p` 标志  
> **解析**：`-p`（`--print`）让 Claude Code 以非交互模式运行，输出结果后退出，不等待用户输入。

---

### Q11 — Batch API 适用性

团队想节省 API 成本。当前有两个工作流：
1. **合并前阻塞检查**：必须在开发者 merge 前完成
2. **每晚技术债报告**：供第二天早上查看

经理建议全部切到 Message Batches API（可节省 50% 成本）。你如何评估？

A. 只把技术债报告改成 batch，合并前检查继续用实时调用  
B. 两个都改 batch，并轮询状态  
C. 两个都保留实时调用  
D. 两个都改 batch，太慢时再 fallback

> **答案：A**  
> **核心考点**：Batch API 不适合阻塞性工作流  
> **解析**：Batch API 最长处理 24 小时，不适合阻塞合并流程。技术债报告延迟容忍度高，适合批处理。

### Q12 — 大型 PR 评审架构

一个 PR 修改了库存模块中的 14 个文件。单次全量审查出现：有些文件反馈细、有些很浅；明显 bug 被漏掉；同一模式在不同文件中得到矛盾判断。应如何**重构评审流程**？

A. 拆成多轮：逐文件分析局部问题，再做一次跨文件集成分析  
B. 要求开发者把大 PR 拆成 3-4 个文件的小 PR  
C. 换上下文更大的模型，一次性继续审  
D. 对整个 PR 跑 3 次，只保留至少 2 次都出现的问题

> **答案：A**  
> **核心考点**：大型代码审查应先逐文件后跨文件，避免注意力稀释  
> **解析**：将大型 PR 审查拆分为两层：(1) 逐文件局部分析，(2) 单独的跨文件集成分析，比一次全量审查更可靠。

## 官方样题核心规律总结

| 题号 | 核心考点 | 关键记忆点 |
|------|---------|-----------|
| Q1 | 程序化 vs 提示 | 有财务后果 → 必须用程序化约束 |
| Q2 | 工具描述质量 | 工具选错 → 第一步改描述 |
| Q3 | 升级校准 | 置信分/情绪 ≠ 可靠升级信号 |
| Q4 | 命令位置 | 团队共享 → `.claude/commands/` |
| Q5 | Plan Mode | 大规模架构 → Plan Mode 优先 |
| Q6 | 路径规则 | 散落文件 → glob 模式 |
| Q7 | 任务分解 | 覆盖不全 → 先看协调者分解 |
| Q8 | 错误传播 | 结构化错误 + 部分结果 + 替代方案 |
| Q9 | 工具范围 | 受限工具处理简单情况，复杂情况升级 |
| Q10 | CI 模式 | CI 必用 `-p` 标志 |
| Q11 | Batch API | 阻塞型任务不适合 Batch |
| Q12 | 代码审查 | 大 PR → 逐文件 + 跨文件两层 |
