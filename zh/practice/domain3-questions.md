## Task 3.1 CLAUDE.md 层级配置（Q1–Q12）

**Q1.** Claude Code 中 CLAUDE.md 的层级体系一般分为几个级别？这些级别的优先级顺序是什么？

A) 仅有全局级和项目级两个级别，优先级为项目级 > 全局级
B) 三个级别：用户全局级、项目级和目录级，优先级为目录级 > 项目级 > 用户全局级
C) 只有目录级和项目级，目录级优先级更高
D) 五个级别包括工作区级、项目级、团队级、目录级和文件级

> **答案：B**
> 解析：Claude Code 支持三层 CLAUDE.md 配置体系，分别位于 ~/.claude/CLAUDE.md（用户全局）、项目根目录 CLAUDE.md（项目级）和各目录下的 CLAUDE.md（目录级）。优先级从高到低为目录级、项目级、用户全局级，即最近的配置优先覆盖远处的配置。

**Q2.** 假设在用户全局 ~/.claude/CLAUDE.md 中定义了编码风格规范，但在某个项目的 CLAUDE.md 中针对同一规范设置了不同的标准。当 Claude Code 在该项目中运行时，会采用哪个规范？

A) 始终采用用户全局规范，确保一致性
B) 采用项目级规范，因为它更接近当前工作上下文
C) 提示用户选择
D) 两个规范都应用，用户可以选择兼容模式

> **答案：B**
> 解析：在 CLAUDE.md 层级体系中，更接近执行上下文的配置文件优先级更高。项目级 CLAUDE.md 覆盖用户全局级别的配置，这样可以支持项目特定的约定和规范。

**Q3.** 在一个大型单体库（monorepo）中，有三个独立的微服务。每个微服务有不同的代码风格和项目约定。应该如何使用 CLAUDE.md 层级来最优地支持这种场景？

A) 在项目根目录创建一个 CLAUDE.md，包含所有三个服务的配置
B) 在每个微服务目录下创建独立的 CLAUDE.md，让目录级配置为各服务提供特定规范
C) 为每个微服务创建独立的 .claude/ 目录
D) 使用全局 CLAUDE.md，通过条件注释区分不同服务

> **答案：B**
> 解析：在 monorepo 场景中，应该充分利用目录级 CLAUDE.md 的灵活性，在每个微服务目录下创建独立的 CLAUDE.md 文件，让 Claude Code 根据当前工作目录加载最相关的配置。这样既保持项目整体一致，又允许各服务的具体约定。

**Q4.** @import 指令在 CLAUDE.md 中的作用是什么？请选择最准确的描述。

A) @import 用于从其他 CLAUDE.md 文件中导入配置片段，支持参数化和条件导入
B) @import 仅用于导入外部文档链接，不能导入配置
C) @import 从其他文件复制内容到当前文件，导入后两个文件内容完全相同
D) @import 是一个已弃用的特性，现代 Claude Code 推荐使用独立的规则文件

> **答案：A**
> 解析：@import 是 CLAUDE.md 中的关键指令，允许从其他 CLAUDE.md 或配置文件导入可重用的配置片段。这支持模块化配置管理和 DRY 原则，避免配置重复。

**Q5.** 假设在项目根目录的 CLAUDE.md 中有 @import 指令引入了一个共享的代码风格配置文件。当在 src/ 目录下创建了目录级 CLAUDE.md 后，这个导入的配置会如何表现？

A) 导入的配置会被目录级 CLAUDE.md 完全覆盖
B) 导入的配置被忽略，只加载目录级 CLAUDE.md
C) 目录级 CLAUDE.md 可以继承并有选择地覆盖导入的配置
D) 两个配置会冲突，导致加载失败

> **答案：C**
> 解析：Claude Code 的继承机制允许目录级配置继承父级的导入配置，并可以选择性地覆盖其中的部分设置。这提供了灵活的配置管理能力。

**Q6.** 在 CLAUDE.md 中定义的配置内容应该主要包含哪些类型的信息？

A) 仅包含 Claude 的系统提示词和行为规范
B) 仅包含项目依赖和构建配置
C) 编码约定、最佳实践指南、项目上下文、工具使用规范等上下文信息
D) 完整的项目架构文档和所有源代码

> **答案：C**
> 解析：CLAUDE.md 应该包含对 Claude Code 理解项目所需的上下文信息，如编码约定、技术栈描述、特殊规范等，而不是项目的完整技术文档或构建配置。

**Q7.** 某个项目中，用户全局 CLAUDE.md 包含了通用的安全最佳实践，项目级 CLAUDE.md 针对特定的安全需求做了扩展。这两个级别的安全规范会如何处理？

A) 只使用项目级规范，全局规范被完全忽略
B) 只使用全局规范，确保所有项目的一致性
C) 项目级规范扩展并可以覆盖全局规范，形成该项目的最终安全政策
D) 两个规范冲突，Claude Code 会抛出错误

> **答案：C**
> 解析：这是 CLAUDE.md 继承和扩展的理想用法。用户可以在全局级定义通用规范作为基础，项目级配置可以在此基础上进行专项扩展，形成项目特定的完整政策。

**Q8.** CLAUDE.md 和 .claude/rules/ 目录中的规则文件有什么本质区别？

A) 两者完全相同，只是存储位置不同
B) CLAUDE.md 是声明式的全局上下文配置，.claude/rules/ 中的规则文件通常针对特定文件模式（glob）进行作用域限制
C) CLAUDE.md 用于配置工具，.claude/rules/ 用于定义命令
D) CLAUDE.md 已被弃用，应该使用 .claude/rules/ 代替

> **答案：B**
> 解析：两者是互补的。CLAUDE.md 提供全局上下文，而 .claude/rules/ 中的规则文件通过 glob 模式可以对特定文件集应用有针对性的规则，提供更细粒度的控制。

**Q9.** 一个项目在开发阶段需要使用宽松的代码风格，但发布前需要切换到严格的代码审查规范。如何在不修改代码库的情况下通过 CLAUDE.md 层级来管理这个需求？

A) 创建两个独立的 CLAUDE.md 文件，每个对应一个阶段
B) 在项目 CLAUDE.md 中使用条件语法指定不同阶段的规范
C) 在用户全局或项目级 CLAUDE.md 中维护两个版本，通过切换来改变有效配置
D) 使用环境变量结合 CLAUDE.md，根据环境加载不同的配置版本

> **答案：D**
> 解析：虽然问题中要求不修改代码库，最优方案是利用 Claude Code 对环境变量的支持，在全局或项目 CLAUDE.md 中根据环境变量条件性地应用不同的规范集，允许灵活切换。

**Q10.** 在什么场景下应该选择在目录级 CLAUDE.md 中定义配置，而不是在项目级 CLAUDE.md 中？

A) 总是应该在项目级 CLAUDE.md 中定义，以保持简洁
B) 当特定目录或子项目有与项目整体不同的约定或特殊需求时
C) 只有在全局 CLAUDE.md 无法满足需求时才使用
D) 从不使用目录级 CLAUDE.md，这会导致配置混乱

> **答案：B**
> 解析：目录级 CLAUDE.md 正是为了支持这种场景而设计的。当某个目录或组件有特殊的编码约定、框架特定的规范或独特的上下文时，在该目录下创建 CLAUDE.md 可以提供精确的上下文控制。

**Q11.** 假设项目中有一个 @import "shared-rules.md" 指令在项目级 CLAUDE.md 中，而 shared-rules.md 本身又包含其他 @import 指令。这种链式导入是否被支持，可能存在什么风险？

A) 不支持，@import 只能引入一级配置文件
B) 支持链式导入，没有特殊风险
C) 支持链式导入，但需要注意避免循环引用，可能导致配置无限递归

> **答案：C**
> 解析：Claude Code 支持嵌套的 @import 指令，但架构师需要注意防止循环依赖。例如，shared-rules.md 不能直接或间接地导入项目级 CLAUDE.md，否则会导致循环依赖和配置加载失败。

**Q12.** 在一个跨团队合作的项目中，基础架构团队维护项目级 CLAUDE.md，应用开发团队需要在各自的代码目录下添加特定规范。最佳实践是什么？

## Task 3.2 自定义斜杠命令（Q13–Q22）

**Q13.** 自定义斜杠命令在 Claude Code 中存储的位置是什么？

> **答案：B**
> 解析：自定义斜杠命令存储在项目的 .claude/commands/ 目录中，每个命令对应一个 Markdown 文件。这样便于版本控制和团队共享。

**Q14.** 斜杠命令中的 $ARGUMENTS 占位符起什么作用？

A) $ARGUMENTS 是一个系统变量，自动获取 Claude 的版本信息

> **答案：B**
> 解析：$ARGUMENTS 是命令文件中的占位符，当用户通过 /command-name arg1 arg2 调用时，arg1 arg2 会替换 $ARGUMENTS，使命令支持参数化。

> 解析：创建自定义命令的标准流程是：在 CLAUDE.md 或项目配置中声明命令存在，然后在 .claude/commands/ 目录下创建对应名称的 Markdown 文件，文件内容为命令的具体指令。

## Task 3.3 glob 规则文件配置（Q23–Q32）

**Q23.** .claude/rules/ 目录中的规则文件主要是用来做什么的？

A) 存储项目的 CI/CD 规则
B) 存储带有 YAML frontmatter 和 glob 模式的规则，对特定文件集应用有针对性的配置
C) 存储数据库迁移规则
D) 存储用户访问控制规则

> **答案：B**
> 解析：.claude/rules/ 中的规则文件使用 YAML frontmatter 定义 glob 模式，使规则能够针对性地作用于匹配该模式的文件。这提供了比全局 CLAUDE.md 更细粒度的控制。

**Q24.** 规则文件中 YAML frontmatter 的标准格式是什么？

A) `[globs]\nsrc/**/*.ts\n[/globs]`
B) ```yaml
---
globs: ["src/**/*.ts", "lib/**/*.ts"]
---
```
C) `// globs: src/**/*.ts`
D) `{globs: ["src/**/*.ts"]}`

> **答案：B**
> 解析：规则文件使用标准的 YAML frontmatter 格式，用三个连字符 --- 包围，globs 是一个包含一个或多个 glob 模式的数组。

**Q25.** 在一个项目中，需要为 TypeScript 源文件（src/）和测试文件（tests/）应用不同的编码规范。应该如何使用 glob 规则？

A) 创建两个规则文件：rules/typescript-src.md（globs: ["src/**/*.ts"]）和 rules/typescript-tests.md（globs: ["tests/**/*.ts"]），分别定义不同的规范
B) 在一个规则文件中使用两个不同的 glob 模式，但这会导致冲突
C) 创建一个通用规则应用于所有 TypeScript 文件
D) 在 CLAUDE.md 中使用条件语句区分

> **答案：A**
> 解析：这是 glob 规则的标准用法。通过创建多个规则文件，每个针对不同的文件模式，可以对不同的代码区域应用不同的规范，提供灵活的配置。

**Q26.** glob 模式 `src/**/*.ts` 和 `src/*/*.ts` 有什么区别？

A) 两者完全相同
B) 前者匹配 src 及其所有子目录下的 .ts 文件，后者只匹配 src 的直接子目录下的 .ts 文件
C) 前者不支持使用，是旧的语法
D) 后者更高效，应该优先使用

> **答案：B**
> 解析：`**` 是递归通配符，匹配任意深度的目录；`*` 只匹配单个目录级别。`src/**/*.ts` 匹配 src/a/b/c/file.ts，而 `src/*/*.ts` 只匹配 src/a/file.ts。

**Q27.** 假设有一个规则文件定义了 globs: ["src/**/*.ts"]，但同时有两个文件 src/utils/helper.ts 和 src/utils/helper.d.ts。如何只让规则应用于 .ts 文件而不是 .d.ts 文件？

A) 在 glob 中使用否定模式 ["src/**/*.ts", "!src/**/*.d.ts"]

B) 将整个 .claude/skills/ 目录提交到 git，团队成员 clone 项目后即可访问所有技能
C) 创建一个中央技能库，团队通过 URL 或包管理器引用
D) 技能必须存储在全局目录，无法在项目级共享

> **答案：B**
> 解析：.claude/skills/ 存储在项目内，完全支持 git 版本控制。团队可以共同维护和演进技能，确保团队内的一致性和可审计性。

**Q59.** 假设一个技能在执行时遇到了错误（如工具调用失败）。在 context: fork 模式下，应该如何处理这个错误？

A) 错误会自动回滚所有修改
B) 技能应该在 Markdown 说明中描述错误处理策略，让 Claude 在遇到错误时进行适当的清理和报告
C) 错误被忽略，执行继续
D) fork 模式下无法处理错误

> **答案：B**
> 解析：虽然 fork 提供隔离，但错误处理需要在技能的设计中明确说明。技能应该描述在不同失败场景下的恢复策略，让 Claude 能够做出正确的决策。

**Q60.** 在设计一个可重用的、高质量的技能时，应该遵循什么最佳实践？

A) 只需要实现功能即可，不需要特殊的结构或文档
B) 明确定义 context（fork 或 inherit）、最小化 allowed-tools 权限、提供详细的 argument-hint、编写清晰的 Markdown 文档、包含错误处理和验证逻辑、设计用于共享和版本控制
C) 技能应该尽可能多地使用所有可用的工具
D) 技能设计没有最佳实践

> **答案：B**
> 解析：这是技能设计的完整最佳实践清单。良好的技能应该是安全的（最小权限）、可理解的（清晰文档）、可维护的（易于测试和调试）和可共享的（支持版本控制）。
