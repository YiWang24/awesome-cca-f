## Task 1.1：智能体循环（Q1–Q9）

---

**Q1.** 你正在实现一个使用 Claude API 的智能体循环。Claude 返回了一个包含 `stop_reason: "tool_use"` 的响应。你的代码应该做什么？

A) 将响应返回给用户，因为 Claude 已经提供了答案  
B) 执行请求的工具，将结果追加到对话历史，然后再次调用 Claude API  
C) 重新发送完全相同的请求，因为响应不完整  
D) 增加 `max_tokens` 并重试，因为模型可能被截断了

> **答案：B**  
> 解析：`"tool_use"` 意味着 Claude 请求调用一个或多个工具。正确流程：执行工具 → 将结果作为 `tool_result` 消息追加到历史 → 再次调用 API。这是智能体循环的核心机制。A 错误：Claude 还没有完成任务。C 错误：重新发送同样的请求会导致无限循环。D 错误：`max_tokens` 与 `tool_use` 的 stop_reason 无关。

---

**Q2.** 以下哪种智能体循环实现模式是**反模式**（应该避免的）？

A) 当 `stop_reason == "tool_use"` 时继续循环，当 `stop_reason == "end_turn"` 时终止  
B) 在迭代之间将工具结果作为 `tool_result` 消息追加到对话历史  
C) 检查 Claude 响应的文本内容中是否包含 "任务完成" 或 "我已经完成" 等短语来决定终止  
D) 在每次工具调用后更新消息历史，使模型能够推断下一步行动

> **答案：C**  
> 解析：解析自然语言信号来判断循环终止是反模式。Claude 的输出是不可预测的自然语言，它可能不会使用这些特定的短语，或者可能在中间步骤使用它们。正确的终止条件是 `stop_reason == "end_turn"`。

**Q3.** 一个开发者实现了如下智能体循环：

```python
for i in range(50):
    response = client.messages.create(model=model, tools=tools, messages=messages)
    if response.stop_reason == "end_turn":
        break
    # 执行工具并更新历史...
```

这个实现的主要问题是什么？

A) 循环次数上限（50次）太低，应该增加到1000次  
B) 使用 `for` 循环而非 `while` 循环是根本性的架构错误  
C) 将任意迭代上限作为主要停止机制是反模式，但这里它是辅助保护，主要停止仍然是 `end_turn` 检查，这是可以接受的  
D) 缺少对 `stop_reason == "tool_use"` 的显式检查

> **答案：C**  
> 解析：这个实现实际上是**可接受**的，因为主要停止条件是正确的 `end_turn` 检查，而 50 次迭代上限只是防止无限循环的安全保护。题目中的反模式是"将任意迭代上限作为**主要**停止机制"——而这里不是。D 也有些道理，但不影响功能正确性，因为当 stop_reason 既不是 end_turn 又可能不是 tool_use 时（如 max_tokens），代码也能处理。

**Q4.** 在智能体循环中，为什么必须在每次迭代之间将工具结果追加到对话历史？

A) 因为 Claude API 有状态，它会自动记住先前的调用  
B) 因为 Claude 模型没有内置内存，每次 API 调用都需要完整的上下文来推断下一步行动  
C) 因为 API 需要验证工具结果的格式  
D) 因为工具结果用于更新模型的权重

> **答案：B**  
> 解析：Claude API 是**无状态**的——每次调用都是独立的。模型没有对先前调用的内置内存。完整的对话历史（包括工具调用和结果）必须在每次请求中传递，这样模型才能基于之前发现的信息推断下一步行动。

**Q5.** 你的智能体调用了一个数据库查询工具，工具返回了一个包含200个字段的大型 JSON 对象。将这个完整对象追加到对话历史的主要风险是什么？

A) Claude 无法解析 JSON 格式  
B) 大型工具结果在上下文中积累，消耗的 token 与其信息价值不成比例，可能导致上下文窗口溢出  
C) Claude 会忽略超过某个大小的工具结果  
D) 数据库响应必须始终转换为自然语言

> **答案：B**  
> 解析：大型工具结果（如每次查询返回40+字段）在上下文中积累，消耗的 token 远超实际所需。最佳实践是在追加到历史之前修剪工具结果，只保留相关字段，从而为更多对话轮次保留上下文空间。

**Q6.** 以下哪个 `stop_reason` 值表示 Claude 已经完成任务并准备好将最终响应呈现给用户？

A) `"tool_use"`  
B) `"max_tokens"`  
C) `"end_turn"`  
D) `"stop_sequence"`

> **答案：C**  
> 解析：`"end_turn"` 表示 Claude 已经完成当前任务，生成了最终响应。这是智能体循环的正常终止信号。`"tool_use"` 表示需要工具调用（继续循环）。`"max_tokens"` 表示输出被截断（错误情况）。`"stop_sequence"` 表示遇到了停止序列。

**Q7.** 模型驱动决策（Claude 基于上下文推断调用哪个工具）与预配置决策树的主要区别是什么？

A) 模型驱动决策总是更慢，因为它需要更多推理时间  
B) 预配置决策树提供确定性的工具调用顺序，而模型驱动决策允许 Claude 根据中间结果动态适应  
C) 模型驱动决策只适用于简单任务，复杂任务必须使用预配置决策树  
D) 预配置决策树和模型驱动决策在功能上是等价的

> **答案：B**  
> 解析：预配置决策树强制执行固定的工具调用顺序（确定性），适合需要严格合规的场景（如金融操作必须先验证身份）。模型驱动决策允许 Claude 根据当前上下文和中间结果动态决定下一步行动（适应性），适合高歧义性的任务。

**Q8.** 一个智能体循环的实现中，开发者没有将 Claude 的 `"assistant"` 消息（包含工具调用请求）追加到历史，只追加了工具结果。这会导致什么问题？

A) API 会拒绝请求，因为消息历史格式无效（assistant 消息和 tool_result 必须配对）  
B) Claude 会忽略工具结果并生成不同的响应  
C) 性能会降低但功能不受影响  
D) 模型会请求执行更多工具来补偿

> **答案：A**  
> 解析：Claude API 要求对话历史格式正确：`tool_result` 消息必须紧跟包含对应 `tool_use` 块的 `assistant` 消息。如果 assistant 消息（包含工具调用请求）没有追加，API 会返回格式错误，因为 `tool_result` 没有对应的 `tool_use` 来关联。

**Q9.** 在实现智能体循环时，哪种做法最能防止在工具执行失败时陷入无限循环？

A) 设置一个非常高的迭代上限（如10000次）  
B) 在工具层面处理错误并向模型返回结构化错误信息，同时作为辅助保护设置合理的最大迭代次数  
C) 在工具失败时立即终止循环并向用户报告错误  
D) 重试失败的工具调用无限次直到成功

> **答案：B**  
> 解析：最佳实践是双重保护：(1) 工具层面的结构化错误响应（让 Claude 能够决定如何处理失败，如重试、换方案或向用户报告）；(2) 合理的最大迭代次数作为安全保护（防止极端情况下的无限循环）。单独依赖任何一种都不够充分。

**Q10.** 在枢纽辐射（Hub-and-Spoke）多智能体架构中，协调智能体的主要职责是什么？

A) 直接执行所有工具调用，子智能体只负责总结结果  
B) 管理所有子智能体间通信、错误处理和信息路由，同时负责任务分解、委托和结果聚合  
C) 仅作为用户和子智能体之间的消息传递层，不做任何决策  
D) 维护一个共享内存池，让所有子智能体都能直接读写

> **答案：B**  
> 解析：协调智能体是枢纽辐射架构的核心，承担：任务分解、子智能体委托、结果聚合、错误处理、信息路由。它是唯一"了解全局"的智能体。子智能体在隔离的上下文中运行，不直接相互通信。

**Q11.** 在多智能体研究系统中，你发现网络搜索子智能体能够正确找到相关文章，但最终报告只覆盖了研究主题的一小部分子集。协调者的日志显示它将主题"气候变化对农业的影响"分解为"对小麦产量的影响"和"对大豆产量的影响"两个子任务。根本原因是什么？

A) 网络搜索子智能体的查询不够广泛  
B) 合成子智能体没有被正确配置  
C) 协调智能体的任务分解过窄——只考虑了作物产量，忽略了气候变化对农业的其他重要维度（如灌溉用水、极端天气事件、种植季节、农业劳动力等）  
D) 子智能体使用了错误的信息源

> **答案：C**  
> 解析：这是任务分解过窄导致不完整覆盖的经典案例（对应官方样题Q7的知识点）。子智能体都正确执行了它们被分配的任务，问题在于协调者的分解本身就是有偏的。气候变化对农业的影响远不止作物产量。

**Q12.** 在多智能体系统中，为什么子智能体间的通信应该通过协调智能体路由，而不是直接进行？

A) 直接通信在技术上不可能  
B) 通过协调者路由确保可观察性、一致的错误处理和受控的信息流，避免子智能体直接依赖彼此状态  
C) 协调者需要收集所有通信记录以供审计  
D) 直接通信会消耗过多的 API 配额

> **答案：B**  
> 解析：通过协调者路由的三大优势：(1) **可观察性**：所有通信经过单一节点，便于监控和调试；(2) **一致错误处理**：协调者统一管理所有失败情况；(3) **受控信息流**：协调者决定哪些信息传递给哪些子智能体，防止信息过载或泄露。

**Q13.** 你的多智能体研究系统产出了一份报告，但在审查后发现某些主题被多个子智能体重复研究，而其他主题没有被任何子智能体覆盖。哪种设计改进最能解决这个问题？

A) 增加更多的子智能体以提高覆盖率  
B) 在协调者的任务分解策略中明确给每个子智能体分配**不同的子主题或来源类型**，最小化重复  
C) 让子智能体在完成后相互通知已经覆盖的主题  
D) 使用共享数据库来跟踪哪些主题已被研究

> **答案：B**  
> 解析：解决重复/遗漏的根本方法是在协调者的任务分解阶段显式分配不重叠的研究范围。例如：子智能体A负责学术论文，子智能体B负责新闻报道，子智能体C负责政策文件。这在设计时就防止了重复。

**Q14.** 协调智能体在收到合成子智能体的第一次输出后，评估发现报告覆盖了主要观点但遗漏了几个关键统计数据。最合适的下一步是什么？

A) 接受输出并向用户呈现，完美是好的敌人  
B) 完全丢弃合成结果并重启整个研究流程  
C) 实现迭代优化循环：识别空白 → 向搜索和分析子智能体委派有针对性的补充查询 → 使用新发现重新调用合成，直到覆盖充分  
D) 直接在协调者中添加缺失的统计数据

> **答案：C**  
> 解析：这正是迭代优化循环的应用场景。协调者不应接受不完整的输出，也不应浪费完整重启的成本。正确做法是基于已有结果识别具体空白，委派针对性的补充研究，然后将新发现融入修订的合成中。

**Q15.** 在多智能体研究系统中，合成子智能体开始直接调用网络搜索API（绕过协调者）来验证其合成过程中的一些声明。这种设计决策的主要问题是什么？

A) 这会增加API成本  
B) 绕过协调者破坏了信息流的可观察性和一致性，使系统难以监控、调试和维护；合成子智能体超出了其角色范围  
C) 网络搜索结果可能与已有研究重复  
D) 这在技术上可行，没有实质性问题

> **答案：B**  
> 解析：所有子智能体通信应通过协调者，这是架构原则。合成子智能体直接进行网络搜索：(1) 破坏了单一观察点；(2) 错误处理变得不一致；(3) 子智能体跨越了其专业边界（合成应该综合，不应搜索）；(4) 调试和追踪变得困难。

**Q16.** 以下哪种场景最适合采用动态自适应的多智能体研究架构（而非固定流水线）？

**Q17.** 你发现协调智能体总是按照"搜索→分析→合成→报告"的固定顺序调用所有四个子智能体，即使对于非常简单的查询也是如此。这种设计的核心问题是什么？

A) 子智能体的数量不合适，应该使用更少的子智能体  
B) 对所有查询强制执行完整流水线而不考虑查询复杂性——协调者应该分析查询需求并动态决定调用哪些子智能体  
C) 固定顺序使系统难以扩展新的子智能体  
D) 这种设计没有问题，一致的流水线简化了系统维护

> **答案：B**  
> 解析：协调智能体的核心价值在于**智能选择**。对于简单事实查询，可能只需要搜索→报告；对于已缓存的主题，可能只需要合成→报告。强制所有查询经过完整流水线浪费了资源，也是对协调者角色的误用。

**Q18.** 多智能体系统中，协调智能体在收到合成子智能体的报告后发现某些领域的覆盖"不够深入"。协调者决定再次调用搜索子智能体获取更多信息。以下哪种方式最能使这次补充搜索有效？

A) 发送与第一次完全相同的搜索任务  
B) 指示搜索子智能体随机搜索更多关键词  
C) 向搜索子智能体提供**具体的覆盖空白**（如"缺少2020-2024年的统计数据"、"没有亚洲市场的数据"），使用有针对性的查询  
D) 增加搜索子智能体的 max_tokens 使其返回更长的结果

> **答案：C**  
> 解析：迭代优化循环的有效性取决于重新委派的**精确性**。协调者应该将评估出的具体空白转换为针对性的补充查询，而不是重复之前的搜索。这使子智能体能够有的放矢地填补具体的知识缺口。

**Q20.** 你的研究协调智能体调用了一个文档分析子智能体，并希望它综合网络搜索的发现。子智能体完成分析后返回了只基于其被分配的初始文档的输出，没有整合网络搜索结果。最可能的根本原因是什么？

A) 子智能体选择忽略了网络搜索结果  
B) 网络搜索结果没有在子智能体的提示中直接提供——子智能体不会自动继承父上下文或在调用之间共享内存  
C) 子智能体需要访问网络搜索工具才能整合那些结果  
D) 协调智能体需要在两个子智能体之间创建共享内存空间

> **答案：B**  
> 解析：这是子智能体上下文隔离的核心问题。子智能体不会自动继承协调者的对话历史。网络搜索结果必须被显式地包含在传递给文档分析子智能体的提示中。如果想要整合的话，协调者必须显式传递所有相关上下文。

**Q21.** 在设计子智能体上下文传递时，以下哪种做法最能保留信息归因（知道每个发现来自哪里）？

A) 将所有发现合并到一个长文本段落中传递给下一个子智能体  
B) 使用结构化数据格式，将内容与元数据（来源URL、文档名称、页码、发布日期）分离  
C) 只传递发现的摘要以节省上下文空间  
D) 用自然语言描述发现并提及来源名称

> **答案：B**  
> 解析：结构化数据格式（如JSON）确保内容和来源元数据绑定在一起，不会在合成过程中分离。纯文本段落（A）使来源信息难以机器处理。仅摘要（C）可能丢失精确的原始来源信息。自然语言提及（D）难以在后续处理中可靠地提取。

## Task 1.6：任务分解策略（Q44–Q51）

---

**Q44.** 提示链（Prompt Chaining）和动态自适应分解（Dynamic Adaptive Decomposition）的主要区别是什么？

A) 提示链只适用于简单任务，动态分解适用于复杂任务  
B) 提示链使用固定的预定义步骤序列；动态自适应分解根据中间发现生成子任务，允许计划随新信息调整  
C) 提示链更昂贵，动态分解更便宜  
D) 两者在功能上等价，只是实现方式不同

> **答案：B**  
> 解析：提示链适用于**可预测**的任务（审查总是包含相同的方面）；动态分解适用于**开放性**任务（在探索前不知道需要哪些步骤）。例如，代码审查（固定：安全、性能、测试、文档）vs 遗留代码库分析（动态：不知道会发现什么架构问题）。

**Q45.** 你需要审查一个包含 20 个文件的 PR。以下哪种任务分解策略最能避免"注意力稀释"问题？

A) 一次性提供所有 20 个文件并要求全面审查  
B) 将文件内容合并后提供，使 Claude 能够更快地处理  
C) 将审查分为两轮：首先逐文件进行本地分析（每次只看一个文件的局部问题），然后进行跨文件整合分析（关注文件间数据流和兼容性）  
D) 随机选择 5 个文件进行深入审查

> **答案：C**  
> 解析：在单个大型提示中处理过多文件会导致注意力稀释——某些文件得到详细分析，其他文件只有表面评论，甚至在不同文件之间出现矛盾的判断。分为逐文件本地分析 + 跨文件整合分析两轮，确保每个文件得到一致深度的关注，同时不遗漏文件间的集成问题。

---

**Q46.** "为遗留代码库添加全面测试"是一个典型的开放式任务。哪种分解方法最合适？

A) 立即开始为每个文件生成测试，逐文件处理  
B) 先映射代码库结构和依赖关系，识别高影响区域，创建优先级计划，然后随着依赖关系的发现动态调整  
C) 使用提示链，将任务分解为：单元测试→集成测试→端到端测试三个固定阶段  
D) 首先询问客户希望测试覆盖哪些文件，然后按顺序处理

> **答案：B**  
> 解析：开放式任务的特征是"在开始之前不知道最重要的关注点是什么"。动态分解（先探索→发现→制定计划→执行）比固定分解更有效，因为它允许优先处理真正重要的部分（高影响区域、核心业务逻辑、有复杂依赖的代码）。

**Q47.** 对于代码库的代码审查，何时应该选择**提示链**（固定序列）而非**动态分解**？

A) 当代码库非常大时  
B) 当审查需要关注固定的、可预测的多个方面时（如安全、性能、测试覆盖率、文档）——每个文件都需要从这些固定角度进行检查  
C) 当只有一个开发者时  
D) 提示链总是优于动态分解

> **答案：B**  
> 解析：提示链适合**结构化的、可预测的多方面任务**。当你知道审查总是需要检查安全漏洞→性能问题→测试覆盖→文档质量，这些步骤是固定的，提示链能提供一致性和可预测性。动态分解适合当你不知道会发现什么的开放性探索。

**Q48.** 你的 CI/CD 流水线运行对 PR 的代码审查。历史数据显示，超过 50 个文件的 PR 会产生不一致的审查结果。最有效的解决方案是什么？

A) 要求开发者将大型 PR 拆分为小型 PR  
B) 升级到具有更大上下文窗口的 Claude 模型  
C) 将审查分为：(1) 每个文件独立分析本地问题；(2) 对相关文件组进行跨文件整合分析  
D) 对大型 PR 跳过自动审查，改用人工审查

> **答案：C**  
> 解析：大型上下文窗口不能解决注意力稀释问题——模型在处理大量信息时仍然会倾向于关注某些部分而忽略其他部分。正确解决方案是任务分解：将大型审查分解为可管理的单元，每次只专注于有限的范围。

**Q49.** 在设计大型代码迁移任务（如将 Java 应用程序从 Spring Boot 2 迁移到 Spring Boot 3）的分解策略时，以下哪种方法最有效？

A) 直接开始迁移，遇到问题时再调整  
B) 先进行全面的代码库分析（Glob/Grep 识别所有需要更改的模式），然后按模块/包分组进行迁移，最后进行集成测试  
C) 将所有文件一次性提供给 Claude，要求全量迁移  
D) 随机迁移文件直到覆盖整个代码库

> **答案：B**  
> 解析：大型迁移任务需要：(1) **先探索**（了解迁移范围和复杂性）；(2) **按模块分组**（防止跨文件依赖导致的不一致）；(3) **验证点**（每个模块迁移后验证功能）。一次性全量处理会导致注意力稀释和遗漏。

**Q50.** 以下哪种情况应该将一个大型任务**拆分为独立的本地分析pass + 跨文件集成pass**（而非单次大型分析）？

A) 当任务只涉及一个文件时  
B) 当审查需要识别文件间不一致性（A文件定义了某函数，B文件以不同的方式使用它）AND 需要对每个文件进行深入的本地分析时  
C) 当开发者想要节省 API 调用成本时  
D) 当任务的时间预算有限时

> **答案：B**  
> 解析：两pass方法在以下情况特别有价值：需要**深度+广度**的结合——每个文件需要深入的本地分析（适合per-file pass），同时需要识别文件间的不一致或接口问题（适合跨文件集成pass）。单一大型分析会在这两个目标之间权衡，导致两者都不够好。

**Q51.** Explore 子智能体（Explore subagent）的主要用途是什么？

A) 替代人工代码审查  
B) 隔离详细的发现输出（详细代码探索、大量读取操作），并将摘要返回主对话，防止主会话的上下文窗口被发现输出填满  
C) 并行处理多个搜索查询  
D) 生成代码文档

> **答案：B**  
> 解析：Explore 子智能体用于隔离**冗长的发现阶段**。例如，探索一个大型代码库需要读取数百个文件，这会产生大量输出。Explore 子智能体处理这个详细探索，只将摘要返回给主会话，使主会话的上下文窗口保持清洁，可以用于后续的实现步骤。

## Task 1.7：会话状态与分叉（Q52–Q60）

---

**Q52.** `--resume <session-name>` 和 `fork_session` 在 Claude Code 中的主要区别是什么？

A) `--resume` 用于恢复崩溃的会话，`fork_session` 用于正常的工作切换  
B) `--resume` 继续同一个对话线程（在之前离开的地方继续）；`fork_session` 从共享基线创建独立分支，允许探索不同的方法而不影响原始会话  
C) `--resume` 适用于所有情况，`fork_session` 是实验性功能  
D) 两者功能相同，只是命令语法不同

> **答案：B**  
> 解析：`--resume` 是线性的——继续同一个对话，在之前停止的地方继续。`fork_session` 是分叉的——从同一个起点创建两个（或多个）独立的分支，每个分支可以探索不同的方法，分支之间互不影响。

**Q53.** 你在昨天的 Claude Code 会话中分析了一个大型代码库，会话中有很多 Grep 和 Read 工具调用的结果。今天你修改了 3 个文件，现在想继续分析。以下哪种方式最合适？

A) 恢复昨天的会话，不提任何新变化——先前的分析仍然有效  
B) 开始一个全新的会话，从头重新分析整个代码库  
C) 恢复昨天的会话，明确告知 Claude 具体哪 3 个文件已修改，请求对这些特定文件的针对性重新分析  
D) 使用 `fork_session` 从昨天的会话分叉，在新分支中处理修改

> **答案：C**  
> 解析：恢复会话时，由于代码已变更，之前的工具结果（文件读取结果）可能已经过时。但大部分分析上下文（架构理解、未修改文件的分析）仍然有效。正确做法是恢复会话 + 通知具体变化，让 Claude 进行针对性的更新分析，而不是浪费资源全量重新分析。

**Q54.** 什么情况下，相对于恢复先前会话，开始一个带有结构化摘要注入的全新会话更合适？

A) 总是——新会话总是比恢复更可靠  
B) 从不——恢复会话总是更好，因为它保留了更多上下文  
C) 当先前会话中的大量工具结果（文件读取结果、搜索结果）已经过时（代码库发生了重大变更）时，先前上下文的存在可能导致基于旧信息的错误推理  
D) 只有当先前会话已经超过24小时时

> **答案：C**  
> 解析：当先前的工具结果（缓存的文件内容、搜索结果）大部分已过时时，恢复会话反而有害——Claude 可能基于过时的信息做出错误推理，但意识不到这个问题。这时，将关键的（仍然有效的）发现提炼为结构化摘要，注入到新会话中，是更可靠的做法。

**Q55.** 你想在完成代码库分析后，同时探索两种不同的重构方案（方案A：将大类拆分为多个小类；方案B：提取共享逻辑到独立模块）。哪种 Claude Code 功能最适合这个场景？

A) `--resume` 两次，分别用于每种方案  
B) `fork_session`：从完成代码库分析的基线分叉出两个独立分支，分别探索方案A和方案B  
C) 在同一会话中顺序探索两种方案  
D) 创建两个全新的独立会话，在每个会话中重新进行代码库分析

> **答案：B**  
> 解析：`fork_session` 正是为这种场景设计的：已有一个共享的基线分析，需要从这个基线出发探索多个不同方向，而不希望这些探索互相影响。与选项D不同，`fork_session` 避免了重复代码库分析的成本。

**Q56.** 在恢复一个会话后，Claude 开始给出与原始分析不一致的答案，并开始提到"典型的设计模式"而非你代码库中的具体实现。这最可能是什么问题？

A) API 连接问题  
B) 上下文窗口退化（Context degradation）——随着会话延长，早期分析的具体细节在上下文中的权重下降，Claude 开始依赖通用知识而非之前发现的具体代码库信息  
C) 模型版本更新导致行为变化  
D) 恢复的会话使用了错误的系统提示

> **答案：B**  
> 解析：这是扩展会话中上下文退化的典型症状。模型的注意力在长上下文中会出现不均匀分布，早期会话中的具体发现可能被新信息"稀释"。解决方案：使用暂存文件（scratchpad）记录关键发现，在后续问题中显式引用这些文件。

**Q57.** 在 Claude Code 会话中，当上下文窗口开始充满大量的发现输出时，哪个命令可以帮助减少上下文使用量？

A) `/clear`  
B) `/reset`  
C) `/compact`  
D) `/summarize`

> **答案：C**  
> 解析：`/compact` 命令用于在扩展的探索会话中减少上下文使用量。它会对现有的上下文进行压缩，保留关键发现但减少冗余的探索输出，从而为后续的代码生成或进一步分析释放上下文空间。

**Q58.** 当你使用 `--resume` 恢复一个会话后，希望 Claude 重新分析已修改的文件。哪种方式效率最高？

A) 告诉 Claude "请重新分析整个代码库，一切都已改变"  
B) 不告诉 Claude 任何变化，让它自己发现  
C) 明确说明哪些特定文件已修改（如"auth.py 和 user_service.py 已修改"），并请求对这些特定文件的针对性重新分析  
D) 结束会话并从头开始

> **答案：C**  
> 解析：精确通知变更范围是最高效的策略。让 Claude 重新分析整个代码库（A）浪费资源——大部分文件没有变化。不通知变化（B）可能导致 Claude 基于过时的缓存信息做出错误推理。针对性通知 + 针对性重新分析是最佳平衡。

**Q59.** 在多智能体研究系统中，当系统崩溃后需要恢复时，哪种设计最能实现可靠的崩溃恢复？

A) 从头重新启动整个研究过程  
B) 依赖 Claude 的内置内存自动恢复状态  
C) 每个智能体在关键检查点将状态导出到已知位置（如JSON清单文件），协调者在恢复时加载这些清单并将状态注入到智能体提示中  
D) 使用数据库存储所有中间结果，恢复时查询数据库

> **答案：C**  
> 解析：结构化状态持久化是可靠的崩溃恢复机制。Claude 没有内置内存（B是错误的）。每个智能体在完成关键步骤时导出状态到文件，协调者维护一个"进度清单"，崩溃后协调者加载清单了解哪些步骤已完成，从中断点恢复，而非重新开始。

**Q60.** 以下哪种场景是使用命名会话（named sessions + `--resume`）的最佳实践用例？

A) 一次性的、不需要后续继续的任务  
B) 每天持续的代码库调查，需要在多个工作日内积累对系统的理解，每天继续上次停止的地方  
C) 单个 bug 修复，有清晰的堆栈跟踪  
D) 快速的代码格式化任务

> **答案：B**  
> 解析：命名会话最适合需要**跨多个工作期间积累理解**的持续性调查任务。通过 `--resume <session-name>` 每天恢复同一个调查会话，Claude 可以逐步建立对系统的深入理解，避免每天重新探索已知内容。一次性任务（A、C、D）不需要会话持久化。
