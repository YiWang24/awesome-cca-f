## Task 4.1 少样本提示与明确标准（Q1–Q12）

**Q1.** 你的企业需要 Claude 将冗长的技术文档总结为简洁的执行摘要。你提供了一个不清晰的指令："写一个好的摘要"。为了确保结果一致性，你应该采取什么行动？

A) 增加 Claude API 调用的温度参数，让模型更具创意
B) 提供 3-5 个不同风格和长度的优秀摘要范例，说明你期望的格式和内容深度
C) 要求 Claude 用符号标记每一个摘要段落
D) 使用不同的模型版本，轮流测试哪个表现最好

> **答案：B**
> 解析：少样本提示通过提供具体的示例来帮助 Claude 理解隐含的偏好。给出 3-5 个范例，展示长度、深度、格式等方面的期望，比模糊的"好的摘要"指令效果好得多。这是建立清晰期望的最直接方式。

**Q2.** 在使用少样本提示时，你正在教 Claude 如何进行情感分析。下面哪种方法最有效地展示你不想要的输出？

A) 在提示末尾添加一条说明："不要输出无关的文本"
B) 提供错误输出的负样例，说明不应该如何分类（例如：错误地将中立意见标记为正面）
C) 多次重复正确的输出示例，直到达到 20 个示例
D) 使用正则表达式限制输出格式

> **答案：B**
> 解析：显式负样例（反面教学）能够帮助 Claude 理解你不想要什么，这与仅提供正面例子一样重要。通过展示常见的误分类错误，你能有效地缩小模型的假设空间，提高分类准确性。

**Q3.** 你正在开发一个内容审核系统。你的原始指令是："检查文本是否包含不当内容"。将其转化为明确的标准，下列哪项最佳？

A) "如果文本包含脏话或仇恨言论，将其标记为不当；如果包含暴力内容描写，将其标记为暴力；如果是关于成人内容的显式描述，将其标记为成人内容。不标记建设性的政治讨论。"
B) "检查文本中的任何有争议的词汇"
C) "使用你的最佳判断来确定什么是不当的"
D) "根据社区标准判断这个文本"

> **答案：A**
> 解析：明确的标准应列举具体的分类类别、包含/排除的定义，以及边界案例。选项 A 提供了清晰的分类定义和反例，使审核决策具有可重复性和一致性，这正是明确标准的核心。

## Task 4.2 降低误判率（Q13–Q24）

**Q13.** 在分类任务中，"假正例"（误报）和"假负例"（漏报）之间有什么区别？

A) 假正例 = 模型说是，实际上不是；假负例 = 模型说不是，实际上是
B) 假正例 = 假数据；假负例 = 真实数据
C) 两者是同义词，用法可互换
D) 假正例用于精准度，假负例用于召回率

> **答案：A**
> 解析：正确的定义对选择优化指标至关重要。假正例（模型错误地预测为正）会导致误报；假负例（模型错误地预测为负）会导致漏报。在选择模型阈值和设计提示时需要理解这种权衡。

**Q14.** 你正在为垃圾邮件检测构建提示。业务需求是最小化假正例（合法邮件被标记为垃圾邮件）。为了实现这一目标，你应该调整什么？

A) 增加模型温度以获得更多创意分类
B) 在提示中明确要求高精准度，降低分类垃圾邮件的阈值（需要更强的证据才能标记为垃圾）
C) 添加更多垃圾邮件示例
D) 使用较低的 top-p 参数

> **答案：B**
> 解析：最小化假正例意味着优化精准度（真正例 / 所有预测正例）。这通过提高阈值要求模型需要更强的证据才能进行正面分类。相反，如果你想最小化假负例，则需要降低阈值，提高召回率。

**Q15.** 你正在为内容审核开发分类器。模型过度触发（将很多合法内容标记为不当）。最可能的原因是什么，应该如何解决？

A) 模型不够聪慧；换一个更强大的模型
B) 缺少负样例（合法内容示例）；添加多个合法内容示例，展示不应该标记为不当的内容
C) 需要更高的温度
D) 使用更严格的 regex 过滤器

> **答案：B**
> 解析：过度触发（高假正例率）通常是因为模型没有足够的反面教学。添加负样例（明确标记为不当的内容示例）能帮助 Claude 理解分类边界，减少误报。

**Q16.** 在精准度（precision）和召回率（recall）之间进行权衡时，下列哪种情况应优先考虑高精准度？

A) 患者风险筛选：你希望捕捉所有潜在的风险病例
B) 欺诈检测：你希望最小化虚假欺诈警报，因为每个警报都很昂贵
C) 医学诊断辅助：你希望确保每个诊断都是准确的
D) 推荐系统：你希望向用户展示尽可能多的相关项目

> **答案：B**
> 解析：欺诈检测中，假正例（合法交易被标记为欺诈）成本很高（用户沮丧、手动审查成本），所以应优先精准度。而在医学筛选中，漏掉一个真实风险（假负例）的成本更高，所以应优先召回率。理解业务成本很关键。

## Task 4.3 结构化输出设计（Q25–Q36）

**Q25.** 何时应该使用 tool_use 来强制结构化输出，而不是在提示中要求 JSON？

A) 总是要求 JSON，tool_use 没有优势
B) 当你需要保证 Claude 100% 输出有效的结构化数据时，使用 tool_use；JSON 格式请求可能导致解析错误或格式不一致

**Q38.** 你正在为一个文章生成任务实现多轮审查。第一轮输出是："人工智能的潜力是无限的"。第二轮应该识别什么问题？

**Q39.** 在多轮架构中，当审查者发现许多问题时应该做什么？

D) 使用常规 API 而不是 Batches

> **答案：B**
> 解析：虽然 Batches API 可以处理大量请求，但分割成合理大小的批处理（如 10,000）提供了更好的可管理性和容错。如果 10,000 个请求的批处理失败，你只需重新提交 10,000 个，而不是 50,000 个。这也防止了超大文件的问题。

**Q54.** 在使用 Batches API 时，如何监测批处理的进度和完成？

A) 轮询 API，定期检查批处理状态，直到它显示"completed"或"failed"
B) 使用 webhook，Batches API 会在完成时发送通知
C) 两者都支持：轮询或 webhook；选择取决于你的架构
D) 无法监测进度

> **答案：C**
> 解析：Batches API 支持两种方式来跟踪完成：轮询（定期查询状态）和 webhook（被动通知）。选择取决于你的架构。对于简单的脚本，轮询可能足够；对于生产系统，webhook 更有效率，但需要一个 Web 端点来接收通知。

**Q55.** 你正在使用 Batches API 进行数据处理。处理中有两个请求失败了。应该怎么办？

A) 整个批处理被认为失败；重新提交所有请求
B) 失败的请求在结果中被标记为失败；你可以识别它们（通过 custom_id）并只重新提交失败的请求
C) 失败的请求被自动重试
D) 失败无法恢复

> **答案：B**
> 解析：Batches API 返回部分结果，标记哪些成功，哪些失败。通过 custom_id，你可以精确地识别哪些请求失败，只重新提交那些。这比重新提交整个批处理更有效率。

**Q56.** 你正在为 Batches API 准备 10,000 个分类请求。每个请求都是类似的，只有输入文本不同。如何最有效地构造请求？

A) 为每个请求使用不同的系统提示
B) 所有请求使用相同的系统提示和模型参数，将输入文本放在每个请求的用户消息中，使用唯一的 custom_id
C) 每个请求应该是完全唯一的
D) 不可能有效地批处理相似的请求

> **答案：B**
> 解析：为了效率，重复的任务应该使用相同的系统提示和参数。只改变用户消息（输入文本）。这允许 Anthropic 的基础设施优化处理，并最大化成本节省。每个请求的 custom_id 允许你跟踪结果。

**Q57.** 使用 Batches API 时，什么时候应该决定不使用它，而改用常规 API？

A) 永远使用 Batches API，它总是更好
B) 如果你需要实时或近实时结果（< 几分钟），或需要使用 tool_use 或多轮对话，使用常规 API；Batches API 适合离线、大规模处理
C) 永远不要使用 Batches API
D) 无法比较

> **答案：B**
> 解析：Batches API 和常规 API 的选择取决于需求。Batches：成本效率，可以等待 24 小时，大规模处理。常规 API：实时响应，tool_use，多轮对话，交互式应用。选择正确的工具取决于你的用例约束。

**Q58.** 你正在使用 Batches API 进行每日报告生成。你有 1,000 个唯一的报告要生成。每个报告需要 2-3 个 Claude 调用（数据收集、初稿、审查）。Batches API 是做这个的好选择吗？

A) 是，Batches API 完美
B) 否，因为每个报告需要多轮，而 Batches API 只支持单轮请求；要么为每个阶段提交分离的批处理，要么对多轮需求使用常规 API
C) 无法用 API 生成报告
D) 应该让人手动做这个

> **答案：B**
> 解析：这突出了一个关键的 Batches API 限制：它不支持多轮对话。你可以变通：为数据收集提交一个批处理，然后为初稿提交另一个批处理，等等。但这更复杂。对于需要密切反馈循环的任务，常规 API 或多轮审查架构可能更合适。

**Q59.** 你正在设计一个大规模的分类系统。预期的负载是每分钟 5,000 个分类请求。Batches API 适合这个吗？

A) 是，Batches API 为任何规模而设计
B) 否，因为 Batches API 被设计为每天或每周的大规模批处理，而不是实时的、持续的流量；对于每分钟 5,000 个请求，使用常规 API 或自定义排队系统可能更适合
C) 应该使用数据库而不是 API
D) 5,000 请求/分钟太多了，任何系统都无法处理

> **答案：B**
> 解析：Batches API 最适合离线、周期性的大规模处理。持续的、持续密集的流量（如实时系统的 5,000/分钟）应该使用常规 API，可能带有排队和速率限制。Batches API 的 24 小时处理窗口对于实时系统来说太长。

**Q60.** 总结一下，以下关于 Message Batches API 的陈述中哪个最准确地描述了它的位置和权衡？

A) Batches API 是所有情况下的最佳选择
B) Batches API 为大规模、离线、非时间敏感的任务提供 50% 的成本节省，代价是没有实时保证，且不支持 tool_use 或多轮；常规 API 适合实时、交互式应用；选择取决于你的延迟和功能需求
C) Batches API 和常规 API 在功能和成本上完全相同
D) 只有少数罕见的用例适合任何 API
