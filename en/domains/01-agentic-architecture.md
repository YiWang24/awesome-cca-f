### Task Statement 1.1: Design and implement agentic loops for autonomous task execution

#### Knowledge of:

- The agentic loop lifecycle: sending requests to Claude, inspecting stop_reason ("tool_use" vs "end_turn"), executing requested tools, and returning results for the next iteration
  - The control signal for the agent loop comes from the `stop_reason` field, not the natural language text generated by the model. `tool_use` means that the model requests to call the tool and the cycle should continue; `end_turn` means that the current round of tasks is completed and the cycle can be terminated. Treating textual content such as "Task Complete" as a termination signal is an anti-pattern because the model output content is unstable.
- How tool results are appended to conversation history so the model can reason about the next action
  - After the tool is executed, the results must be appended to the message history in `tool_result` format (role is user). If this step is skipped, the model will not be able to perceive the tool execution results in the next round and will make inferences based on expired status, which may lead to repeated calls to the same tool or broken logic links.
- The distinction between model-driven decision-making (Claude reasons about which tool to call next based on context) and pre-configured decision trees or tool sequences
  - Model-driven decision-making allows Claude to dynamically select tools based on context, which is suitable for open and highly ambiguous tasks; the pre-configured decision tree solidifies the steps in the code, which is suitable for scenarios that require deterministic compliance (such as financial operations). The core trade-off is flexibility and predictability, and different business scenarios have different optimal solutions.

#### Skills in:

- Implementing agentic loop control flow that continues when stop_reason is "tool_use" and terminates when stop_reason is "end_turn"
  - When implementing control flow, a complete branch judgment should be made for `stop_reason`: the `tool_use` branch is responsible for executing the tool and writing back the results, and the `end_turn` branch terminates the loop. Iteration caps (such as `max_iterations`) can only be used as protective caps, not as primary stopping conditions.
- Adding tool results to conversation context between iterations so the model can incorporate new information into its reasoning
  - After each round of tool execution, the system needs to first append the assistant's complete response (including the `tool_use` block) to the history, and then append `tool_result` (must contain matching `tool_use_id`) with the user role. Wrong order or mismatched IDs will cause API errors or model inference errors.
- Avoiding anti-patterns such as parsing natural language signals to determine loop termination, setting arbitrary iteration caps as the primary stopping mechanism, or checking for assistant text content as a completion indicator
  - Parsing model output text (e.g. detecting "Task Complete") to control loop termination is unreliable because the model wording is slightly different each time. A fixed round cap is also insufficient, as tasks may not be completed when the rounds are exhausted. `stop_reason` is the only structured termination basis and should be used as the main control signal.

### Task Statement 1.2: Orchestrate multi-agent systems with coordinator-subagent patterns

#### Knowledge of:

- Hub-and-spoke architecture where a coordinator agent manages all inter-subagent communication, error handling, and information routing
  - In the Hub-and-spoke architecture, the coordinator is responsible for all routing, aggregation and error handling; there is no direct communication between sub-agents. This design ensures that all information flows have a traceable central node, which facilitates auditing, debugging, and unified error policies.
- How subagents operate with isolated context—they do not inherit the coordinator's conversation history automatically
  - Child agents run in independent sessions and do not automatically inherit the coordinator's conversation history. Each time the Task tool is called, the task goal, scope constraints, existing findings, output format, and source metadata must be explicitly passed into the subagent's prompt.
- The role of the coordinator in task decomposition, delegation, result aggregation, and deciding which subagents to invoke based on query complexity
  - The coordinator's responsibilities include task decomposition, dynamic delegation, result aggregation, coverage gap identification, and error handling decisions. Simple queries can be responded directly without starting a complete multi-agent pipeline; only complex or large-scale tasks are worthy of starting dedicated sub-agents on demand.
- Risks of overly narrow task decomposition by the coordinator, leading to incomplete coverage of broad research topics
  - Too fine a granularity of task decomposition will lead to a coverage gap: some subtopics are missed because there are no corresponding subagent assignments. This is a common quality defect in multi-agent systems. The root cause is not an error in the execution of a single subtask, but the omission of important areas in the task boundary design.

#### Skills in:- Designing coordinator agents that analyze query requirements and dynamically select which subagents to invoke rather than always routing through the full pipeline
  - The coordinator should dynamically select a processing path based on the query complexity: simple queries are answered directly without delegating to subagents; only complex multi-domain research tasks generate specialized search, analysis, and synthesis subagents on demand. Forcing every request through the full pipeline adds latency and wastes resources.
- Partitioning research scope across subagents to minimize duplication (e.g., assigning distinct subtopics or source types to each agent)
  - The scope of work of sub-agents should be mutually exclusive and complementary, divided by topic, source type, time frame or evidence type to avoid duplication of investigations into the same areas. Overlapping scopes not only waste resources, but also produce conflicting conclusions during the aggregation stage and increase coordination costs.
- Implementing iterative refinement loops where the coordinator evaluates synthesis output for gaps, re-delegates to search and analysis subagents with targeted queries, and re-invokes synthesis until coverage is sufficient
  - After completing the initial synthesis, the facilitator should evaluate the output for thematic gaps. After discovering the gap, the search or analysis sub-agent should be re-commissioned with specific targeted questions to trigger the synthesis again. This iterative refinement cycle is the core mechanism for ensuring the quality of research coverage.
- Routing all subagent communication through the coordinator for observability, consistent error handling, and controlled information flow
  - All sub-agent outputs are uniformly aggregated to the coordinator, making it easy to track the source of each conclusion, error propagation path, and data boundaries. Decentralized peer-to-peer communication breaks observability and complicates debugging and governance.

### Task Statement 1.3: Configure subagent invocation, context passing, and spawning

#### Knowledge of:

- The Task tool as the mechanism for spawning subagents, and the requirement that allowedTools must include "Task" for a coordinator to invoke subagents
  - Task tool is the only mechanism to generate child agents. The coordinator's `allowedTools` field must contain `"Task"`, otherwise the child agent cannot be called - even if the system prompt explicitly states that it can, the runtime will fail due to lack of tool permissions.
- That subagent context must be explicitly provided in the prompt—subagents do not automatically inherit parent context or share memory between invocations
  - The child agent does not share the parent session memory, and each Task call is a new session. The prompt passed in should contain all the context needed to complete the task: target, scope constraints, known facts, source metadata, failure records (if any), and desired output format.
- The AgentDefinition configuration including descriptions, system prompts, and tool restrictions for each subagent type
  - AgentDefinition sets role boundaries for each type of subagent through descriptions, system prompts, and `allowed_tools` fields. The search agent only holds search tools, and the synthesis agent only holds formatting tools to prevent role drift and unauthorized calls.
- Fork-based session management for exploring divergent approaches from a shared analysis baseline
  - `fork_session` Create independent branches from a common analysis baseline, suitable for scenarios where multiple mutually exclusive solutions need to be explored in parallel (such as refactoring strategy A vs B). Alternately reasoning about different solutions in the same context can easily cause context pollution between solutions.

#### Skills in:

- Including complete findings from prior agents directly in the subagent's prompt (e.g., passing web search results and document analysis outputs to the synthesis subagent)
  - The synthetic subagent does not have session history access permission to search the subagent. The coordinator needs to pass all search results, document analysis output, source URLs, confidence levels, and known restrictions into the prompt of the synthesis agent to ensure that the synthesis is based on complete information.
- Using structured data formats to separate content from metadata (source URLs, document names, page numbers) when passing context between agents to preserve attribution
  - When delivering context, content (excerpts, conclusions) should be separated from metadata (source URL, document name, page number, date), and structured formats (such as JSON) should be used instead of natural language paragraphs. Loss of the source chain after abstraction is a common traceability flaw in multi-agent synthesis.
- Spawning parallel subagents by emitting multiple Task tool calls in a single coordinator response rather than across separate turns
  - Parallel execution requires the coordinator to issue multiple Task tool calls in a single response. If initiated sequentially across multiple turns, even logically independent tasks become serial execution, and total latency becomes the sum of each subtask.
- Designing coordinator prompts that specify research goals and quality criteria rather than step-by-step procedural instructions, to enable subagent adaptability
  - Facilitator's prompt should describe the research objectives, quality thresholds (e.g. at least one source for each conclusion), and delivery format rather than step-by-step instructions. Goal-oriented prompts enable the sub-agent to adaptively adjust the search strategy based on intermediate findings.

### Task Statement 1.4: Implement multi-step workflows with enforcement and handoff patterns

#### Knowledge of:- The difference between programmatic enforcement (hooks, prerequisite gates) and prompt-based guidance for workflow ordering
  - The prompt instructions are probabilistic constraints, and the model has a non-zero probability of not complying with them. Hooks and precondition gating are deterministic constraints that ensure execution order through code-level blocking. Workflow sequences involving compliance, permissions, and finance must use programmatic controls.
- When deterministic compliance is required (e.g., identity verification before financial operations), prompt instructions alone have a non-zero failure rate
  - When the business process requires deterministic compliance (such as identity verification before refunding), relying solely on prompt instructions will produce a non-zero failure rate. Precondition checks must be implemented at the code level to block high-risk tool calls until the pre-steps are completed and valid results are returned.
- Structured handoff protocols for mid-process escalation that include customer details, root cause analysis, and recommended actions
  - Human handoff protocols should contain a structured summary (Customer ID, Issue Type, Root Cause, Attempted Actions, Recommended Actions) rather than the raw conversation transcript. Humans usually don't have time to read the entire transcript when taking over, and a structured handoff allows them to immediately understand the case and continue processing.

#### Skills in:

- Implementing programmatic prerequisites that block downstream tool calls until prerequisite steps have completed (e.g., blocking process_refund until get_customer has returned a verified customer ID)
  - Programmatic preconditions are implemented in the tool call interception hook (PreToolUse): check whether there is a verified customer ID, if not met, return an error message and indicate which step needs to be completed first, preventing downstream tool calls from continuing to execute.
- Decomposing multi-concern customer requests into distinct items, then investigating each in parallel using shared context before synthesizing a unified resolution
  - The customer's composite request (such as billing, returns, and account issues) should be broken down into independent issues, investigated in parallel with shared context, and then synthesized into a unified response. Serial processing increases response latency and misses the efficiency benefit of parallel subagents.
- Compiling structured handoff summaries (customer ID, root cause, refund amount, recommended action) when escalating to human agents who lack access to the conversation transcript
  - Standard fields for structured handoff summaries include: customer ID, issue type, root cause analysis, refund amount, sequence of actions attempted, recommended next steps, and urgency. When the human takeover system and the dialogue system do not share history, this summary is the only source of context.

### Task Statement 1.5: Apply Agent SDK hooks for tool call interception and data normalization

#### Knowledge of:

- Hook patterns (e.g., PostToolUse) that intercept tool results for transformation before the model processes them
  - `PostToolUse` Hook is triggered after the tool execution is completed and before the model processing results. It can convert the format of the results (such as unifying the timestamp format, semanticizing the status code, and standardizing the field names). This pre-processing is deterministic and does not rely on the model to understand different formats on its own.
- Hook patterns that intercept outgoing tool calls to enforce compliance rules (e.g., blocking refunds above a threshold)
  - `PreToolUse` Hook is triggered before the tool call is issued. It can check whether the parameters violate business rules (such as the refund amount exceeds the threshold) and block the call. After interception, an alternative path (such as upgrading to manual approval) should be returned instead of just returning an error message.
- The distinction between using hooks for deterministic guarantees versus relying on prompt instructions for probabilistic compliance
  - Hooks provide deterministic guarantees and are suitable for rules that will have financial, compliance or security consequences if violated; prompt instructions provide probabilistic guidance and are suitable for low-risk scenarios such as style preferences and best practices. The two are not interchangeable, the selection criterion is the cost of failure.

#### Skills in:

- Implementing PostToolUse hooks to normalize heterogeneous data formats (Unix timestamps, ISO 8601, numeric status codes) from different MCP tools before the agent processes them
  - Different MCP tools may return heterogeneous formats (Unix timestamp vs ISO 8601, numeric status code vs string), `PostToolUse` Hook unifies the format before these data enter the model context, eliminating the risk of misunderstandings caused by format differences in the model.
- Implementing tool call interception hooks that block policy-violating actions (e.g., refunds exceeding $500) and redirect to alternative workflows (e.g., human escalation)
  - After the tool call is intercepted, the Hook should return a description containing an alternative path (such as "The amount exceeds the automatic approval limit and has been transferred to the manual approval process"), so that the model can explain the next step to the user instead of staying in an error state.
- Choosing hooks over prompt-based enforcement when business rules require guaranteed compliance
  - Any business rules that will have financial, privacy or compliance consequences if violated should be protected through Hooks or back-end verification, not just written in system prompts. The rules in the prompt rely on model compliance, and there is a probability of failure; code-level checks are enforced in the tool execution path.

### Task Statement 1.6: Design task decomposition strategies for complex workflows

#### Knowledge of:- When to use fixed sequential pipelines (prompt chaining) versus dynamic adaptive decomposition based on intermediate findings
  - The fixed-sequence prompt chain is suitable for workflows with known steps and stable input and output at each step (for example: security review first → then performance review → finally readability review); dynamic decomposition is suitable for open investigation tasks, because the next step depends on the current discovery and cannot be fixed in advance.
- Prompt chaining patterns that break reviews into sequential steps (e.g., analyze each file individually, then run a cross-file integration pass)
  - Split large code reviews into two passes: the first step performs local analysis file by file, focusing on issues within a single file; the second step independently performs cross-file integration analysis, focusing on interface contracts, data flows, and inter-module dependencies. Analyzing all documents at once dilutes attention and reduces the quality of review of each document.
- The value of adaptive investigation plans that generate subtasks based on what is discovered at each step
  - The value of dynamic decomposition is that the intermediate discovery of tasks (such as the discovery of critical dependencies, unexpected code structures) can trigger the generation of new subtasks. Supplementary testing of legacy systems, investigation of accident root causes, and exploration of unknown code bases all fall into this category. If all steps are solidified in advance, the critical path will be missed.

#### Skills in:

- Selecting task decomposition patterns appropriate to the workflow: prompt chaining for predictable multi-aspect reviews, dynamic decomposition for open-ended investigation tasks
  - The choice of task decomposition method should match the uncertainty of the workflow: tasks with predictable steps use prompt chains to ensure execution order; tasks with steps that depend on intermediate discovery use dynamic decomposition to maintain adaptability. Incorrectly using a fixed pipeline for open tasks can lead to missing important findings.
- Splitting large code reviews into per-file local analysis passes plus a separate cross-file integration pass to avoid attention dilution
  - Two-stage code review architecture: the first stage analyzes each file independently to capture local bugs; the second stage specifically checks for cross-file issues (interface compatibility, state sharing, permission propagation, circular dependencies). The two stages must be executed independently, and mixing them will lead to in-depth treatment of problems in both dimensions.
- Decomposing open-ended tasks (e.g., "add comprehensive tests to a legacy codebase") by first mapping structure, identifying high-impact areas, then creating a prioritized plan that adapts as dependencies are discovered
  - Recommended decomposition process for open tasks: first use Grep/Glob to build a code base structure map, then identify high-impact areas (core modules, high-frequency calling paths), then dynamically generate prioritized sub-task plans based on dependencies and risks, and continuously adjust as new dependencies are discovered during execution.

### Task Statement 1.7: Manage session state, resumption, and forking

#### Knowledge of:

- Named session resumption using --resume <session-name> to continue a specific prior conversation
  - `--resume <session-name>` Restores the session with the specified name, retaining the original conversation history and context. Ideal for continuing active investigations across work cycles to avoid re-exploring content that has already been analyzed. The premise is that the tool results in the old context have not expired.
- fork_session for creating independent branches from a shared analysis baseline to explore divergent approaches
  - `fork_session` Create independent session branches from a common analysis baseline, suitable for scenarios where different implementations need to be compared in parallel (such as refactoring strategy A vs strategy B, test framework X vs framework Y). Independent branches prevent the reasoning contexts of different solutions from interfering with each other.
- The importance of informing the agent about changes to previously analyzed files when resuming sessions after code modifications
  - After resuming the session, the model still holds the old tool call results and file content cache. If the code is modified between two sessions, the specific changed files must be clearly informed and the affected paths must be reanalyzed. Otherwise, the model will reason based on outdated information and draw wrong conclusions.
- Why starting a new session with a structured summary is more reliable than resuming with stale tool results
  - When a large number of tool results in the old session are out of date (such as after a large-scale refactoring, dependency upgrade), hard recovery may be more risky than opening a new session, because the model will mix the expired state with new information. New Session + Structured Snippets provide a cleaner, more reliable starting point.

#### Skills in:

- Using --resume with session names to continue named investigation sessions across work sessions
  - Long-term investigations across work cycles can use named sessions to preserve context, but before resuming you should evaluate whether the tool results in the session are still valid. The session name itself does not guarantee the timeliness of the information. You need to actively confirm whether the context reflects the current code base status.
- Using fork_session to create parallel exploration branches (e.g., comparing two testing strategies or refactoring approaches from a shared codebase analysis)
  - When comparing Plan A/B in parallel, two independent branches should be forked from the same analysis baseline, each plan should be explored separately, and the coordinator or developer should finally compare the output results. Alternately reasoning about two alternatives in the same session pollutes the context, making the evaluation of each alternative impure.
- Choosing between session resumption (when prior context is mostly valid) and starting fresh with injected summaries (when prior tool results are stale)
  - The criterion for restoring vs. creating is the factual validity of the old context: when file modifications are small (only affecting a few known files), it is more efficient to restore and notify the changes; after large-scale file modifications or dependency upgrades, new sessions + structured summaries can prevent outdated information from affecting subsequent reasoning.
- Informing a resumed session about specific file changes for targeted re-analysis rather than requiring full re-exploration
  - Provide a detailed list of changes (which files were modified, which interfaces were changed) after resuming the session, so that the model can be re-analyzed in a targeted manner instead of re-exploring the entire code base. This preserves the original investigation results while ensuring that new information is processed correctly.

## Task 1.1: The Agentic Loop

### What this tests

Task 1.1 is not really about "calling tools" — it’s about treating the Claude API response as a structured control protocol. The exam draws a sharp line between reliable loops (driven by `stop_reason`, `tool_use_id`, `tool_result`) and brittle ones (parsing natural language, capping iterations arbitrarily, checking text output for completion phrases).

The agentic loop is the smallest closed loop in any autonomous system: Claude decides the next step based on context, and the host program executes tools and writes results back into history. State consistency breaks the moment you miss an assistant `tool_use` block, skip a concurrent `tool_use`, or return a `tool_result` referencing the wrong `tool_use_id`. Think of it as "model reasons, program controls" — Claude decides *what* to do, structured fields ensure the loop remains recoverable, auditable, and terminable.

### Core concepts

The agent loop is the core mechanism that allows Claude to complete multi-step tasks autonomously. It drives the loop through `stop_reason`:

```
Send request → Claude response → check stop_reason
                                │
              ┌─────────────────┼─────────────────┐
              ▼                 ▼                  ▼
         "tool_use" "end_turn" "max_tokens" / Others
              │                 │                  │
    Execute tool, append results Return final response Handle truncation/pause
              │
         Return to top of loop
```

### stop_reason Full enumeration

| Value | Meaning | Loop action |
| ------------------ | ---------------------------------- | -------------------------------------------------- |
| `"tool_use"` | Claude requests to call the tool | **Continue**: Execute the tool → append results → call again |
| `"end_turn"` | Task completed normally | **Termination**: Return final text response |
| `"max_tokens"` | `max_tokens` limit reached, response truncated | **Processing**: Increase `max_tokens` or process in batches |
| `"stop_sequence"` | Preset stop sequence encountered | **Termination**: Processed according to business logic |
| `"pause_turn"` | The tool call is suspended externally (streaming scenario) | **Resume**: Continue after resolving the cause of the suspension |

> It’s easy to get confused here: when `max_tokens` is truncated, `stop_reason` is `"max_tokens"` instead of `"tool_use"`, and the loop cannot simply continue.

### Correct implementation

```python
messages = [{"role": "user", "content": user_query}]

while True:
    response = client.messages.create(
        model="claude-opus-4-6",
        max_tokens=4096,
        tools=tools,
        messages=messages
    )

    if response.stop_reason == "end_turn":
        # Complete normally, extract text response
        final_text = next(
            (b.text for b in response.content if hasattr(b, "text")), ""
        )
        break

    if response.stop_reason == "max_tokens":
        # Response is truncated and needs to be dealt with (increase max_tokens or redesign)
        raise ValueError("Response truncated, please increase max_tokens")

    if response.stop_reason == "tool_use":
        tool_use_blocks = [b for b in response.content if b.type == "tool_use"]

        # Step 1: Append Claude's response (containing tool_use block) to history
        messages.append({"role": "assistant", "content": response.content})

        # Step 2: Execute each tool call
        tool_results = []
        for tool_call in tool_use_blocks:
            result = execute_tool(tool_call.name, tool_call.input)
            tool_results.append({
                "type": "tool_result",
                "tool_use_id": tool_call.id, # must match the id of tool_use
                "content": result # String or list of content blocks
            })

        # Step 3: Append tool results to history (role is user)
        messages.append({"role": "user", "content": tool_results})
```

### tool_result Content format

```python
# Simple string result
{
    "type": "tool_result",
    "tool_use_id": "toolu_abc123",
    "content": "Order status: Shipped, expected to arrive in 3 days"
}

# Structured content blocks (supports images)
{
    "type": "tool_result",
    "tool_use_id": "toolu_abc123",
    "content": [
        {"type": "text", "text": "Here are screenshots:"},
        {"type": "image", "source": {"type": "base64", "media_type": "image/png", "data": "..."}}
    ]
}

# When tool execution fails (set is_error: true)
{
    "type": "tool_result",
    "tool_use_id": "toolu_abc123",
    "content": "Database connection timeout",
    "is_error": True # Claude will adjust subsequent behavior accordingly
}
```

### Anti-Pattern

| Anti-Patterns | Problems |
|------------------------------------------------ | ----------------------------------------------------------------------------------------------- |
| Detect "task completed" in the text to terminate the loop | Claude's output is unpredictable and will be missed or misjudged |
| `for i in range(20):` serves as the **primary** stopping mechanism | Any upper limit is unreliable; `max_iterations` can only be used as a **safe backup**, the primary control must be `stop_reason` |
| Does not append tool results to message history | Claude cannot sense tool results and will retry the same tool repeatedly |
| Forgot to append assistant response to history | Tool result does not correspond to tool_use, API returns 400 error |
| Multiple tool_use only process the first one | Claude can initiate multiple tool calls in one response, all must be processed |

### Model-driven vs. pre-configured decision trees

| Method | Description | Applicable scenarios |
|----------------|------------------------------------------------|--------------------------------|
| **Model driven** | Claude decides independently which tool to call and when to stop | Highly ambiguous tasks require flexible judgment |
| **Pre-configured decision trees** | Fixed order of tool calls (programmed control flow) | Financial/medical processes requiring deterministic compliance |

### When not to use multi-agent architecture

Multiple agents increase complexity, and a single agent is better in the following situations:

- Tasks can be completed within a single context window
- Steps must be strictly serial and interdependent
- Latency-sensitive workflow (each subagent call adds scheduling, context-transfer, and aggregation overhead)
- Unacceptably difficult to debug

Do not judge latency only by whether work can run in parallel. Parallel subagents can reduce total time for independent subtasks, but for small tasks, strictly serial flows, or user-facing interactions, the fixed overhead of spawning and aggregating subagents can exceed the benefit.

## Task 1.2: The Hub-and-Spoke Multi-Agent Pattern

### What this tests

Task 1.2 tests orchestration boundaries. The coordinator isn't a simple router that "runs all sub-agents" — it decides based on query complexity whether to decompose, who gets which pieces, what context to pass, how to merge conflicting results, and when to re-delegate after spotting coverage gaps. The hub-and-spoke emphasis exists because this architecture centralizes communication, error handling, observability, and information flow on a single coordinator, preventing untraceable implicit dependencies between sub-agents.

In practice, multi-agent is a good fit when the research scope is large, the context would exceed limits, tool permissions need isolation, or parallel specialized analysis is required. The main risk isn't "not enough agents" but wrong splitting granularity: too narrow and you'll miss areas, too overlapping and you'll produce conflicting conclusions. Good design has the coordinator define the coverage map first, sub-agents work within complementary scopes, then the coordinator does coverage checking, conflict resolution, and targeted re-delegation.

### Hub-and-Spoke architecture

```
                    ┌──────────────────┐
                    │   Coordinator    │ ← owns routing, errors, information flow
                    └────────┬─────────┘
              ┌──────────────┼──────────────┐
              ▼              ▼              ▼
      ┌───────────┐  ┌───────────┐  ┌───────────┐
      │  Search   │  │ Analysis  │  │ Synthesis │
      │   agent   │  │   agent   │  │   agent   │
      └───────────┘  └───────────┘  └───────────┘
```

**Coordination agent core responsibilities:**

- **Task Decomposition**: Split complex tasks into parallel subtasks
- **Delegation**: Dynamically decide which sub-agents to call and what context to pass
- **Result aggregation**: merge sub-agent output and handle conflicts
- **Error handling**: Unified management of failure situations (retry/use partial results/upgrade)
- **Coverage Check**: Identify task coverage gaps and re-delegate

### Easy to step into pitfalls

**The child agent will not automatically inherit the context of the coordinating agent! **
The child agent runs in a completely independent session, and the context must be passed explicitly to the child agent's prompt.

**Too narrow task decomposition** will lead to incomplete research coverage. Such questions are often induced by seemingly reasonable splits:

- Wrong: Breaking down "AI's impact on creative industries" only into "digital art, graphic design, photography" (missing music, writing, movies, etc.)
- Right: Complete scope is covered during decomposition, and the coordinator is responsible for identifying and filling gaps

### Multi-agent vs single-agent decision-making

```
Task Characteristics Assessment:
├─ Parallelizable + beyond a single context window → multi-agent
├─ Requires specialized sub-agents (different tool sets) → multiple agents
├─ Single linear process + sufficient context → single agent
└─ High latency sensitivity → Prioritize single agent
```

## Task 1.3: Subagent Spawning and Context Transfer

### What this tests

Task 1.3 is about getting subagent spawning right. Two things are tested: first, that `Task` is itself a tool, so the coordinator's `allowedTools` must include `"Task"` — a system prompt saying "You can dispatch subtasks" doesn't give runtime permission. Second, that child agents don't inherit parent session history. Information found in the parent session will not automatically appear in the child agent's context.

In practice, context quality determines output quality. The prompt handed to a subagent needs to include the goal, scope constraints, known facts, source metadata, output format, and quality standards. Too little and the agent will re-explore work already done; too little attribution metadata and the synthesis step loses traceability. For parallel work, the coordinator must issue multiple `Task` calls within a single response — issuing them across separate turns forces serial execution. For comparing approaches, fork independent branches from a shared baseline rather than reasoning about both paths in the same session.

### Task tool mechanism

If the coordination agent wants to call the sub-agent, **`allowedTools` must contain `"Task"`**:

```python
coordinator = AgentDefinition(
    system_prompt="You are the research coordination agent...",
    allowed_tools=["Task", "search_web"], # must contain"Task"
)
```

If there is no `"Task"` in `allowedTools`, the coordinator will not be able to generate child agents - even if the system prompt says it can.

### Generate sub-agents in parallel

Issue multiple `Task` calls in a single response to achieve parallel execution (rather than multiple rounds of serialization):

```python
# Parallelism: Multiple Task calls are included in a single response from the coordinator
response_content = [
    Task(prompt="Search for the latest papers on topic A", context=shared_context),
    Task(prompt="Search statistics for topic B", context=shared_context),
    Task(prompt="Search Case Studies for Topic C", context=shared_context),
]
```

## Task 1.4: Multi-step Workflow Enforcement

### What this tests

Task 1.4 is about distinguishing "prompt suggestions" from "programmatic enforcement." If a sequence is simply a best practice (e.g., "summarize before answering"), a prompt instruction is usually fine. But if the sequence involves finance, identity, permissions, privacy, or compliance (e.g., "verify customer before refunding"), you must use hooks, prerequisite gating, or backend validation. The phrase "non-zero failure rate" that appears in official materials means: prompts alone cannot provide deterministic compliance.

In practice, multi-step workflows also need to handle composite requests and human handoffs. A user request may contain multiple concerns — split them into independent items for parallel investigation, then synthesize a unified response. When escalating to a human, don't just say "please handle this manually." Provide a structured handoff: customer ID, issue type, root cause, attempted actions, refund amount, recommended action, and urgency level. This gives the human agent enough context to continue without reading the entire conversation transcript.

### Programmatic enforcement vs prompt guidance

| Method | Reliability | Applicable scenarios |
| ---------------------------------------- | -------------------------- | ---------------------------------- |
| **Programmatic Enforcement** (Hooks, Precondition Gating) | **Certainty** (100%) | Identity verification, mandatory compliance process before financial operations |
| **Guidance Tips** | **Probabilistic** (with non-zero failure rate) | Best practice recommendations, preferences, style guidelines |

> Remember this sentence: When deterministic compliance is required (such as identity verification before refunding), **only prompt instructions have a non-zero failure rate** and programmatic enforcement must be used. When "non-zero failure rate" appears in a question, it is usually a reminder not to rely solely on prompts.

### Implementation of programmatic prerequisites

```python
def intercept_tool_call(self, tool_name: str, tool_input: dict):
    """
    Tool call interception hook (PreToolUse)
    Force get_customer to be called before lookup_order or process_refund
    """
    protected_tools = ["lookup_order", "process_refund"]

    if tool_name in protected_tools:
        if not self.verified_customer_id:
            # A blocking message is returned and Claude will understand that authentication is required first
            return {
                "error": "You must first call get_customer to verify the customer's identity",
                "required_action": "Please call the get_customer tool first"
            }
    return None # None = Allow to continue
```

## Task 1.5: Agent SDK Hooks

### What this tests

Task 1.5 covers two hook positions: before tool calls go out (`PreToolUse`) and before tool results reach the model (`PostToolUse`). `PreToolUse` handles policy enforcement — blocking refunds that exceed a threshold, rejecting operations on unverified accounts, intercepting dangerous commands. `PostToolUse` handles data normalization — converting timestamps, status codes, currency units, and field names from heterogeneous MCP tools into a consistent format before Claude sees them.

The practical value of hooks is moving deterministic rules from prompts to code. Claude can understand "don't refund more than $500" but can't guarantee compliance every time. A hook guarantees it — the offending call won't execute, and the hook can return an alternative path (escalate to human, request additional approval). The exam judgment rule: if a rule violation would have financial, compliance, security, or data quality consequences, use hooks or backend validation instead of system prompt instructions.

### Hook type and triggering time

| Hook type | Trigger timing | Common uses |
| ------------- | -------------------------- | ---------------------------------- |
| `PreToolUse` | **before** tool calls are issued | Compliance rule execution, parameter verification, access control || `PostToolUse` | **After tool execution**, before model processing | Data format standardization, result filtering, logging |

### PostToolUse: Data normalization

Solve the problem of inconsistent return formats from different tools (deterministic guarantee, do not rely on the model to handle it by itself):

```python
def post_tool_use_hook(tool_name: str, tool_result: dict) -> dict:
    """
    Normalize tool results before entering model context
    Deterministic: Handle format differences by itself without relying on the model
    """
    if tool_name == "get_customer":
        # Unix timestamp → ISO 8601
        if "created_at" in tool_result and isinstance(tool_result["created_at"], int):
            tool_result["created_at"] = datetime.fromtimestamp(
                tool_result["created_at"]
            ).isoformat()

        # numeric status code → semantic string
        status_map = {1: "active", 2: "suspended", 3: "closed"}
        if isinstance(tool_result.get("status"), int):
            tool_result["status"] = status_map.get(tool_result["status"], "unknown")

        # Currency amount: cents → yuan
        if "balance_cents" in tool_result:
            tool_result["balance_yuan"] = tool_result.pop("balance_cents") / 100

    return tool_result
```

### PreToolUse: Access Control and Compliance

```python
def pre_tool_use_hook(tool_name: str, tool_input: dict) -> dict:
    """
    Compliance check before tool invocation
    Return {"blocked": True, ...} to prevent the call; {"blocked": False} to allow it
    """
    if tool_name == "process_refund":
        amount = tool_input.get("amount", 0)
        if amount > 500:
            return {
                "blocked": True,
                "redirect_to": "escalate_to_human",
                "reason": f"The refund amount ¥{amount} exceeds the automatic approval limit ¥500"
            }

        if not self.customer_verified:
            return {
                "blocked": True,
                "reason": "The customer's identity has not been verified and the refund operation is refused."
            }

    return {"blocked": False} # Allow to continue
```

### Hooks vs Prompt Guide Comparison

| Aspects | Hooks | Prompts |
| -------- | -------------------- | -------------------- |
| Reliability | Certainty (100%) | Probabilistic (non-zero failure rate) |
| Applicable scenarios | Compliance, verification, format conversion | Style, preferences, suggestions |
| Implementation location | Code layer (interception execution) | System prompts |

## Task 1.6: Task Decomposition Strategy

### What this tests

Task 1.6 tests how to match decomposition method to task uncertainty. Fixed-sequence prompt chaining works when the steps are known and each step's inputs/outputs are stable — for example, file-by-file review followed by a cross-file integration pass. Dynamic adaptive decomposition fits open-ended tasks where the next step depends on what you discover: supplementing tests in a legacy codebase, diagnosing unknown failures, or researching a broad topic. Hard-coding an open investigation into a fixed pipeline is the wrong approach, as is leaving the determination entirely to the model.

In practice, good decomposition reduces cognitive load while maintaining complete coverage. Large code reviews should run file-by-file first (to avoid attention dilution), then run a separate cross-file pass focused on interface contracts, data flows, state sharing, and integration risks. Open tasks should map structure first, identify high-impact areas, set priorities, then dynamically adjust the plan as dependencies and risks surface.

### Two decomposition methods

| Method | Description | Applicable scenarios |
| ----------------------- | ---------------------------------- | --------------------------------------------------------------- |
| **Prompt chain (fixed order)** | Predefined sequential steps, the output of each step is the input of the next step | Multi-faceted review of known processes (such as code review: security → performance → readability) |
| **Dynamic Adaptive Decomposition** | Dynamically generate next subtasks based on intermediate findings | Open investigation (legacy code base exploration, unknown field research) |

**Open task decomposition process:**

```
1. Mapping structure (Glob/Grep traversal to understand the code base overview)
   ↓
2. Identify high-impact areas (dynamically adjust priorities based on the findings in the first step)
   ↓
3. In-depth analysis (assign sub-agents according to priority)
   ↓
4. Coverage check (the coordinator verifies whether there are any missing areas)
```

**Large code reviews avoid attention dilution:**

- **Phase 1**: File-by-file local analysis (only focusing on local issues of one file at a time)
- **Phase 2**: Independent cross-file integration analysis (focusing on inter-file data flow, interface compatibility)
- The two stages must run independently and cannot be mixed into the same context

## Task 1.7: Session State Management

### What this tests

Task 1.7 is about context freshness in long-running tasks. `--resume <session-name>` fits when the prior context is mostly still accurate. `fork_session` fits when you want to explore different solutions from a shared baseline. A new session with injected summaries fits when old tool results have gone stale. A common exam trap is treating "restore context" as always better — stale tool results can actually be more dangerous than no context at all.

In practice, after resuming a session, you must explicitly tell the agent which files, configurations, or external states have changed so it can run targeted re-analysis. Don’t assume the model will realize on its own that cached findings have expired. The decision rule is straightforward: if the old context is still accurate, resume and report the differences; if large portions have expired, start fresh and inject a structured summary of completed work, key findings, pending items, and current file state.

### Three session operations

| Operation | Command/Mechanism | Usage Scenario |
| ------------------------ | -------------------------------- | ---------------------------------- |
| **Resume named session** | `claude --resume <session-name>` | Continue the previous conversation, the context is still valid || **Session Forking** | `fork_session` | Explore different solutions from the same baseline (A/B comparison) |
| **New session + snippet injection** | New session + structured snippet | Previous tool results have expired and the code has been significantly modified |

### Select decision tree

```
Is the previous context still valid?
├─ yes → --resume resume session
│ └─ Any files changed?
│ ├─ No → Continue directly
│ └─ Yes → Inform the specific changed files and request targeted re-analysis
│ (Don’t let the model discover changes on its own – inefficient and likely to be missed)
└─ No (code has been significantly modified / tool results expired)
    └─ New session + inject structured snippets
        The summary should include: completed work, key findings, open issues

Need to compare two options? → fork_session creates a parallel branch
```

### Digest injection format

When you need to open a new session to continue previous work, inject a structured summary instead of the full history:

```python
structured_summary = """
## Summary of previous work
- Completed: analyzed the auth and payment modules
- Key finding: JWT validation missing expiration check (auth.py:45)
- Pending: notification module has not been analyzed yet
- Known dependencies: payment → notification (event-driven)
"""
messages = [
    {"role": "user", "content": structured_summary + "\nPlease continue to analyze the notification module"}
]
```

## Domain 1 deep dive: making sense of agent architecture and orchestration

Domain 1 carries the most exam weight because it tests something harder than "knowing Claude can call tools" — it tests whether you can embed Claude into a reliable software system. Think of it as a three-tier structure: the single-agent loop (Task 1.1) forms the base, multi-agent orchestration (Tasks 1.2–1.3) adds coordination, and production-level reliability (Tasks 1.4–1.7) adds enforcement, decomposition, and session management. These aren't separate topics; they're layers of the same system expanding from a minimal loop to complex production workflows.

### 1. The agent loop is a state machine, not a while loop

The most common beginner mistake is writing `while True` and exiting when the model outputs "Task Completed." The exam explicitly flags this as an anti-pattern. A correct loop is driven by the structured state in `stop_reason`:

- `tool_use` — the model needs to act; pause, execute the tool, append the result, call again
- `end_turn` — the model has finished this turn; return the response
- `max_tokens` — output was truncated; do not continue as if `tool_use` — investigate the token budget or task scope
- `pause_turn`, `stop_sequence` — handle per business logic

Each `stop_reason` value demands a specific handling branch, not a fallthrough. Similarly, the tool result append sequence is part of the state machine: save the full assistant response (including all `tool_use` blocks), execute every `tool_use`, then append all `tool_result` messages using the matching `tool_use_id` before calling the model again. Deviating from this order corrupts conversation history.

### 2. When model reasoning ends and programmatic control begins

Domain 1 repeatedly tests this boundary: let Claude decide, or let code enforce. Claude should drive semantic judgments — splitting research topics, selecting search terms, deciding whether more evidence is needed. Code should control compliance, permissions, finance, and identity steps that cannot be wrong. When you see "verify customer before refunding," "block access without authorization," or "escalate if amount exceeds threshold" in an exam scenario, the answer points toward hooks, prerequisite gates, or backend validation — not prompt instructions.

This boundary also shapes multi-agent design. The coordinator can let Claude decide whether search, analysis, and synthesis agents are needed. But Claude shouldn't be allowed to bypass approvals, and sub-agents shouldn't hold high-risk tools outside their role. Coverage checks and delivery conditions should be structurally defined: each claim needs at least one source, each gap must be flagged, each failure must carry error context.

### 3. The coordinator is an editor-in-chief, not a router

Hub-and-spoke is often misread as "the coordinator sends tasks to sub-agents." The real responsibilities are: assess query complexity, decide whether multi-agent is needed, define non-overlapping task scopes, deliver context, merge results, identify coverage gaps, handle errors, and decide whether to iterate. The coordinator is part project manager, part quality gatekeeper.

Narrow task decomposition is a common exam trap. A coordinator asked to study "AI's impact on creative industries" that only decomposes into "digital art, graphic design, photography" has a coverage failure — music, writing, film, games, and advertising are all missing. The bug isn't in the sub-agents; it's in the task scope design. Correct decomposition defines the full coverage boundary first, splits along mutually exclusive dimensions, then checks for gaps after the initial synthesis.

Information must flow only through the coordinator. Direct sub-agent-to-sub-agent communication breaks observability: when the final report contains an error, you can no longer trace whether the problem was incomplete search, a document parsing mistake, synthesis misreading, or an unresolved source conflict.

### 4. Sub-agent context must be explicitly assembled

"The child agent doesn't inherit the parent's context" is one of the most-tested concepts in Domain 1. Many wrong designs pass vague instructions like "continue the research" to a search agent — but the sub-agent has never seen "the research." It only sees this Task prompt.

A complete delegation includes: task goal, scope boundaries, existing findings, prohibited duplicates, expected output format, source attribution requirements, quality standards, and failure reporting format. A synthesis agent receiving only "Write a summary" will produce a generic result; the same agent receiving structured search findings with claim–source mappings and a required output schema produces traceable, mergeable output.

For parallel work, issue multiple Task calls in a single coordinator response. Issuing them across separate turns forces serial execution — total latency becomes the sum of each subtask. Multiple calls in one response run concurrently.

### 5. Human handoff is a structured handover, not a transcript dump

When escalating to a human, handing over a raw conversation transcript is the worst option. Humans don't have time to re-read everything, and transcripts contain irrelevant tool output. A proper handoff includes: customer ID, issue summary, verified facts, root cause, attempted actions, amounts or order numbers, recommended next steps, and urgency. A good handoff lets the human agent start with "I see your order ORD-123 needs a manual refund approval because it exceeds the automatic limit" — not "Can you explain your issue again?" Exam answers that say "switch to human" without preserving structured context are usually wrong.

### 6. Hooks provide deterministic guardrails

Hooks serve two distinct purposes: `PostToolUse` normalizes heterogeneous tool outputs (Unix timestamps, ISO dates, numeric status codes, string statuses) before Claude sees them; `PreToolUse` intercepts policy-violating actions before they execute — refunds above a threshold, operations on unverified accounts, dangerous commands.

After blocking a call, return an alternative path, not just an error. "The amount exceeds the automatic limit and has been routed to manual approval" lets Claude explain the next step to the user; a bare error leaves the workflow stuck.

### 7. Task decomposition must match task uncertainty

Fixed prompt chains work when every step is known and each step's inputs/outputs are stable: classify → extract → validate → output. Dynamic adaptive decomposition works for open investigations where the next step depends on what you find: legacy codebase testing, root cause analysis, broad market research. Hard-coding an open investigation into a fixed pipeline is a common architecture mistake.

For large code reviews, use two passes. The file-by-file pass focuses on local bugs, exception handling, and boundary conditions. The cross-file integration pass focuses on data flow, interface contracts, state consistency, and permission propagation. Cramming all files into one context dilutes attention, produces missed detections, and often yields contradictory findings.

### 8. Session resume and fork: freshness over continuity

`--resume` preserves context but carries a risk: tool results from prior sessions can become stale after code changes, dependency upgrades, or external state changes. Always tell the resumed agent which files changed so it can re-examine them. When most of the prior context has expired, starting fresh with a structured summary is safer than resuming with stale findings baked in.

`fork_session` is for parallel exploration from a shared baseline — one branch takes the minimal-change approach, another tries an architectural rewrite. Each explores independently and avoids contaminating the other's reasoning. Never reason about two mutually exclusive approaches in the same session context.

### 9. A five-step framework for Domain 1 questions

1. Does the task require a tool loop? → Focus on `stop_reason`, `tool_use`, `tool_result`
2. Does it need multiple agents? → Check for parallelizability, scope size, specialized tools, context limits
3. Are there compliance or financial constraints? → Programmatic enforcement, not prompt guidance
4. Is context being transferred? → Must be explicitly assembled in the subagent's prompt
5. Is there long-task state? → Evaluate whether resume, fork, or fresh-session-with-summary is safest

Wrong answer patterns to watch for: natural language as loop control signals, prompts used as hard constraints, sub-agents implicitly sharing context, every request forced through the full pipeline, global abort on single failures, resuming sessions without notifying file changes.

The mental model that ties Domain 1 together: Claude handles semantic judgment, open exploration, and synthesis — code handles state machines, permissions, compliance thresholds, and error boundaries. When you can draw the request flow as a diagram — user request → coordinator → sub-agents with explicit context → normalized tool results → coordinator coverage check → synthesize or escalate — you can spot which link is missing in any exam question.

## Pre-exam checklist

1. **stop_reason complete set**: `end_turn` (complete), `tool_use` (adjusting tool), `max_tokens` (truncation), `stop_sequence`, `pause_turn`
2. **Loop Termination**: Always check `stop_reason == "end_turn"`, do not parse text
3. **Tool result format**: `type: "tool_result"` + `tool_use_id` (must match) + `content`
4. **is_error flag**: Set `"is_error": True` when tool execution fails, and Claude will adjust its behavior accordingly.
5. **Multiple tool_use**: A single response can contain multiple tool_use blocks, and **must all be processed**
6. **Sub-agent context**: not automatically inherited, must be passed explicitly
7. **Task tool**: `allowedTools` must contain `"Task"` to generate sub-agent
8. **Parallel execution**: Issue multiple Task calls in a single response (not multiple rounds of serialization)
9. **Programmatic vs Tip**: Key compliance must use programmatic constraints (Hooks)10. **PostToolUse**: Tool result normalization (format conversion deterministic > let the model handle it by itself)
11. **Task decomposition**: Use dynamic decomposition for open tasks, and use prompt chains for known processes.
12. **Session Recovery**: Inform the changed files, don’t let the model discover it on its own
